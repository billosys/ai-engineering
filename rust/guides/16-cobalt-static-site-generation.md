# Cobalt Static Site Generation

Guidelines for building static websites with Cobalt, a Rust-based static site generator that transforms Markdown and Liquid templates into deployable HTML. Covers CLI workflows, content authoring, Liquid templating, configuration, asset pipelines, deployment, and extending Cobalt programmatically with the Rust API.


## CB-01: Install Cobalt via cargo install

**Strength**: MUST

**Summary**: Install Cobalt using `cargo install cobalt-bin`; the crate name is `cobalt-bin`, not `cobalt`.

```console
// Bad - wrong crate name
$ cargo install cobalt

// Good - correct crate name on crates.io
$ cargo install cobalt-bin

// Good - alternative: install via pre-built binary
$ curl -LSfs https://raw.githubusercontent.com/crate-ci/gh-install/master/v1/install.sh \
    | sh -s -- --git cobalt-org/cobalt.rs --crate cobalt

// Good - alternative: macOS via Homebrew
$ brew install cobalt
```

**Rationale**: The crate published on crates.io is `cobalt-bin`, but the installed binary is invoked as `cobalt`. Using the wrong crate name will either fail or install an unrelated package.

**See also**: CB-02, CB-03

## CB-02: Initialize Sites with cobalt init

**Strength**: MUST

**Summary**: Run `cobalt init` to scaffold a new site with the standard directory structure, example content, and configuration file.

```console
// Good - initialize in a new directory
$ mkdir my-site && cd my-site
$ cobalt init

// Good - initialize in a named sub-directory
$ cobalt init myBlog

// Verify the scaffolded structure
$ ls
_cobalt.yml  _defaults/  _includes/  _layouts/  index.liquid  posts/
```

**Rationale**: `cobalt init` creates the `_cobalt.yml` configuration file and all conventional directories (`_layouts`, `_includes`, `_defaults`, `posts`). Manually creating these is error-prone; init provides a working starting point that can be immediately served or built.

**See also**: CB-03, CB-05

## CB-03: Understand the Directory Structure

**Strength**: MUST

**Summary**: Cobalt uses underscore-prefixed directories for site generation resources; the source hierarchy is mirrored in the output.

```
.
|- _cobalt.yml          # Site-wide configuration
|- _layouts/            # Layout templates (e.g., default.liquid)
|-   default.liquid
|- _includes/           # Reusable template snippets
|-   header.liquid
|- _data/               # Data files (YAML, JSON, TOML)
|-   movies.json
|- _sass/               # Sass/SCSS partials for import
|-   base.scss
|- _site/               # Build output (generated by cobalt build)
|- _defaults/           # Templates for cobalt new
|-   pages.md
|-   posts.md
|- posts/               # Blog posts (no underscore prefix!)
|-   cats.md
|- _drafts/             # Draft posts (excluded from build by default)
|-   dogs.md
|- _syntaxes/           # Custom Sublime Text syntax definitions
|- index.liquid         # Site pages live at the root level
```

**Rationale**: Files and folders prefixed with `_` are "hidden" from Cobalt and not copied to the output. Understanding this convention prevents accidentally hiding content or placing resources in the wrong location. Note that the posts directory is `posts/` (no underscore), unlike some other static site generators.

**See also**: CB-05, CB-19

## CB-04: Build and Serve the Site

**Strength**: MUST

**Summary**: Use `cobalt build` for production output and `cobalt serve` for local development with live reload.

```console
// Good - build for deployment (outputs to _site/)
$ cobalt build

// Good - local development with file watching
$ cobalt serve
# => Server Listening on http://localhost:1024

// Good - serve with drafts visible
$ cobalt serve --drafts

// Good - serve on a custom port
$ cobalt serve --port 3000

// Good - verbose build for debugging
$ cobalt build -vv
```

**Rationale**: `cobalt build` writes to the configured destination directory (`_site/` by default) and exits -- use this for deployment artifacts. `cobalt serve` builds to a temporary directory, watches for changes, and runs a local server -- use this during development. Confusing the two leads to stale output or unexpected temporary files.

**See also**: CB-02, CB-07

## CB-05: Create Content with cobalt new

**Strength**: SHOULD

**Summary**: Use `cobalt new` to create pages and posts from `_defaults` templates with proper slug-based filenames.

```console
// Good - create a page in the current directory
$ cobalt new "About Me"
# Creates: about-me.md

// Good - create a post in the posts directory
$ cobalt new "Cats Around the World" --file posts
# Creates: posts/cats-around-the-world.md

// Good - create a post with a custom filename
$ cobalt new "Cats Around the World" --file posts/cats.md
# Creates: posts/cats.md
```

**Rationale**: `cobalt new` generates slug-based filenames and populates frontmatter from templates in `_defaults/`. This ensures consistent naming conventions and complete frontmatter. The `--file` flag controls whether the content is created as a page or a post -- the target directory determines the content type.

**See also**: CB-06, CB-07, CB-08

## CB-06: Distinguish Pages from Posts

**Strength**: MUST

**Summary**: Posts are pages that live in the `posts/` directory and gain additional features; all posts are pages, but not all pages are posts.

```yaml
# A page: any file with a template extension in the source directory
# File: about.md
---
title: About
layout: default.liquid
---
Welcome to my site.

# A post: a page that lives in the posts/ directory
# File: posts/2024-01-15-my-first-post.md
---
title: My First Post
published_date: "2024-01-15 12:00:00 +0000"
layout: posts.liquid
---
This is a blog post with date ordering, RSS support, and draft workflow.
```

**Rationale**: Pages are the fundamental content unit -- any file with a recognized template extension (`.md`, `.liquid`) that is not hidden (no `_` or `.` prefix). Posts are a specialized subset that live in the configured posts directory and gain draft support, chronological ordering, RSS/JSON feed generation, and pagination. Placing a file outside `posts/` means it loses these features.

**See also**: CB-07, CB-08, CB-09

## CB-07: Use the Draft/Publish Workflow for Posts

**Strength**: SHOULD

**Summary**: Posts start as drafts and must be published before they appear in production builds.

```console
// Create a new post (starts as a draft)
$ cobalt new "My New Post" --file posts

// Preview drafts locally
$ cobalt serve --drafts

// Publish when ready (sets published_date, clears is_draft)
$ cobalt publish posts/my-new-post.md

// Build for production (drafts excluded by default)
$ cobalt build
```

```yaml
# Two ways to mark a post as a draft:

# 1. Frontmatter flag
---
is_draft: true
---

# 2. Place in the _drafts/ directory (is_draft inferred automatically)
# File: _drafts/work-in-progress.md
```

**Rationale**: `cobalt publish` performs three actions: sets `is_draft: false`, sets `published_date` to today, and (if in `_drafts/`) moves the file to `posts/`. By default, the date is prepended to the filename (`2024-01-15-my-new-post.md`) for chronological ordering. This workflow prevents unfinished content from appearing in production builds.

**See also**: CB-05, CB-06

## CB-08: Write Frontmatter Correctly

**Strength**: MUST

**Summary**: Place optional YAML metadata between `---` delimiters at the top of content files to control page processing.

```yaml
# Good - complete frontmatter with common fields
---
title: My First Blogpost
published_date: "2016-01-01 21:00:00 +0100"
layout: posts.liquid
is_draft: false
permalink: /blog/{{slug}}
excerpt_separator: "\n\n"
categories:
  - Tutorials
tags:
  - rust
  - cobalt
data:
  author: Jane Doe
---
Content goes here, below the closing delimiter.

# Good - minimal frontmatter (all fields have defaults)
---
title: About
---

# Bad - missing closing delimiter (entire file parsed as YAML)
---
title: Broken Page
Content accidentally treated as YAML...
```

**Rationale**: Frontmatter controls title, layout, permalink, draft state, categories, tags, and custom data. All fields have sensible defaults (`format: Raw`, `is_draft: false`, `permalink: path`), and some values (slug, title, published_date, format, is_draft) can be inferred from the filename. Forgetting the closing `---` delimiter is a common error that causes the entire file to be parsed as YAML.

**See also**: CB-06, CB-09, CB-19

## CB-09: Configure Permalinks for Clean URLs

**Strength**: SHOULD

**Summary**: Use permalink templates in frontmatter to control URL structure; omit file extensions for pretty URLs.

```yaml
# Default: path style (mirrors source hierarchy)
---
permalink: path
---
# File: some/page.md -> URL: /some/page.html

# Good - custom date-based blog URLs
---
permalink: /blog/{{year}}/{{month}}/{{slug}}
---
# Creates: /blog/2024/01/my-post/index.html (pretty URL)

# Good - category-based URLs
---
title: Corgi
categories:
  - Animals
  - Dogs
permalink: /{{categories}}/{{slug}}
---
# Creates: /animals/dogs/corgi/index.html

# Good - set permalink defaults site-wide in _cobalt.yml
# _cobalt.yml:
posts:
  default:
    permalink: /blog/{{year}}/{{month}}/{{slug}}
```

**Rationale**: Permalink templates must start with `/` and support Liquid-style `{{variable}}` placeholders. When the permalink omits a file extension, Cobalt generates "pretty URLs" by creating `index.html` inside a folder. Setting defaults in `_cobalt.yml` ensures consistent URL patterns across the site without repeating the template in every file's frontmatter.

**See also**: CB-08, CB-19

## CB-10: Understand the Content Processing Pipeline

**Strength**: MUST

**Summary**: Every page goes through four ordered steps: Liquid evaluation, Markdown conversion, layout wrapping, and file writing.

```
Source file (e.g., about.md)
  |
  v
Step 1: Evaluate Liquid template expressions ({{ }}, {% %})
  |
  v
Step 2: Convert Markdown to HTML (for .md files only)
  |
  v
Step 3: Wrap in layout template (if layout: specified in frontmatter)
  |
  v
Step 4: Write to destination directory as .html
```

```yaml
# The format field controls step 2
---
format: Raw     # Default for .liquid files; skip Markdown conversion
---

---
format: Markdown  # Inferred for .md files; enable Markdown conversion
---
```

**Rationale**: The ordering matters: Liquid is evaluated *before* Markdown conversion. This means Liquid expressions can generate Markdown content, but Markdown cannot generate Liquid expressions. Understanding this pipeline is essential for debugging template issues -- if output looks wrong, determine which step is responsible.

**See also**: CB-11, CB-12, CB-13

## CB-11: Use Liquid Syntax Correctly

**Strength**: MUST

**Summary**: Use `{{ }}` for outputting values and `{% %}` for control flow; they are not interchangeable.

```html
<!-- Good - output tags render values -->
<title>{{ page.title }}</title>
<p>Published: {{ page.published_date }}</p>

<!-- Good - logic tags for control flow (no output) -->
{% if page.is_draft %}
  <span class="draft">DRAFT</span>
{% endif %}

<!-- Good - filters transform output via pipe -->
<h1>{{ page.title | upcase }}</h1>
<p>{{ page.published_date | date: "%B %d, %Y" }}</p>

<!-- Bad - using logic tags where output is needed -->
{% page.title %}  <!-- This executes nothing, produces no output -->

<!-- Bad - using output tags for control flow -->
{{ if page.is_draft }}  <!-- This outputs the literal text -->
```

**Rationale**: Cobalt uses a Rust implementation of Liquid (`liquid` crate v0.26.11), not Ruby Liquid. The syntax is largely compatible, but some filters or behaviors may differ. The `{{ }}` and `{% %}` delimiters serve fundamentally different purposes and confusing them is a common source of template errors.

**See also**: CB-12, CB-13, CB-14

## CB-12: Create Layouts for Shared Page Structure

**Strength**: MUST

**Summary**: Store layout templates in `_layouts/` and insert page content with `{{ page.content }}`.

```html
<!-- _layouts/default.liquid -->
<!DOCTYPE html>
<html>
  <head>
    {% include "head.liquid" %}
  </head>
  <body>
    <header>
      {% include "header.liquid" %}
    </header>
    <main>
      {{ page.content }}
    </main>
    <footer>
      {% include "footer.liquid" %}
    </footer>
  </body>
</html>
```

```yaml
# Reference the layout from a page's frontmatter
---
title: About
layout: default.liquid
---
```

**Rationale**: Layouts define the HTML skeleton shared across pages. The rendered page content is injected at `{{ page.content }}`, and the excerpt is available as `{{ page.excerpt }}`. Forgetting `{{ page.content }}` in a layout causes page body content to silently disappear. Layouts must be in `_layouts/` to be found by Cobalt.

**See also**: CB-13, CB-14

## CB-13: Use Includes for Reusable Template Components

**Strength**: SHOULD

**Summary**: Store reusable template fragments in `_includes/` and pull them in with `{% include "filename.liquid" %}`.

```html
<!-- _includes/head.liquid -->
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>{{ page.title }} | {{ site.title }}</title>

<!-- _includes/nav.liquid -->
<nav>
  <a href="/">Home</a>
  <a href="/about.html">About</a>
  <a href="/blog/">Blog</a>
</nav>

<!-- Usage in a layout or page -->
{% include "nav.liquid" %}
```

**Rationale**: Includes enable DRY template composition -- shared navigation, headers, footers, and metadata blocks are defined once and reused across layouts and pages. Includes have access to the same variables as the template that includes them. Unlike layouts (which wrap content), includes are fragments inserted inline.

**See also**: CB-12, CB-14

## CB-14: Access Template Variables with Dot Notation

**Strength**: MUST

**Summary**: Cobalt provides three global variable objects: `site`, `page`, and `collection`, accessed via dot notation.

```html
<!-- Site variables (from _cobalt.yml) -->
<title>{{ site.title }}</title>
<link rel="canonical" href="{{ site.base_url }}{{ page.permalink }}">

<!-- Page variables (from frontmatter and build context) -->
<h1>{{ page.title }}</h1>
<time>{{ page.published_date | date: "%Y-%m-%d" }}</time>

<!-- Custom page data (from frontmatter data: field) -->
<span>By {{ page.data.author }}</span>

<!-- Collection variable (iterate over posts) -->
{% for post in collection.posts %}
  <a href="/{{ post.permalink }}">{{ post.title }}</a>
{% endfor %}
```

**Rationale**: `page.content` and `page.excerpt` are only available in layouts, not in page content itself -- trying to access them in a page body silently produces nothing. `page.data` contains user-defined frontmatter data, while `site.data` contains data from the `_data` directory. Understanding variable scoping prevents subtle template bugs.

**See also**: CB-11, CB-12, CB-15

## CB-15: Iterate Over Posts in Templates

**Strength**: SHOULD

**Summary**: Use the `collection.posts` variable to list posts; use the `paginator` object when pagination is active.

```html
<!-- Good - simple post listing without pagination -->
{% for post in collection.posts %}
<article>
  <h2><a href="/{{ post.permalink }}">{{ post.title }}</a></h2>
  <time>{{ post.published_date | date: "%B %d, %Y" }}</time>
</article>
{% endfor %}

<!-- Good - with pagination active (replaces collection) -->
{% for post in paginator.pages %}
<article>
  <h2><a href="/{{ post.permalink }}">{{ post.title }}</a></h2>
</article>
{% endfor %}

<!-- Pagination navigation -->
{% if paginator.previous_index %}
  <a href="/{{ paginator.previous_index_permalink }}">Previous</a>
{% endif %}
{% if paginator.next_index %}
  <a href="/{{ paginator.next_index_permalink }}">Next</a>
{% endif %}
```

**Rationale**: When pagination is active on a page, the `paginator` object replaces `collection` for accessing posts. Using `collection.posts` on a paginated page will not work as expected. Posts are sorted by `published_date` in descending order by default (`posts.order: Desc`).

**See also**: CB-14, CB-23

## CB-16: Use Liquid Control Flow Tags

**Strength**: SHOULD

**Summary**: Use `if`/`elsif`/`else`, `unless`, `case`/`when`, and `for` blocks for template logic.

```html
<!-- Conditional rendering -->
{% if page.is_draft %}
  <span class="badge">Draft</span>
{% elsif page.categories contains "Featured" %}
  <span class="badge">Featured</span>
{% else %}
  <span class="badge">Published</span>
{% endif %}

<!-- Unless (inverse if) -->
{% unless page.tags == empty %}
  <div class="tags">
    {% for tag in page.tags %}
      <span>{{ tag }}</span>
    {% endfor %}
  </div>
{% endunless %}

<!-- Case/when -->
{% case page.data.status %}
  {% when "active" %}
    <span class="green">Active</span>
  {% when "archived" %}
    <span class="gray">Archived</span>
{% endcase %}

<!-- For loop with forloop object -->
{% for post in collection.posts %}
  {{ post.title }}{% unless forloop.last %}, {% endunless %}
{% endfor %}
```

**Rationale**: Cobalt's Liquid provides the standard control flow blocks (if, unless, case, for) plus the `forloop` object with properties like `first`, `last`, `index`, and `length`. These are essential for building dynamic templates that adapt to content structure.

**See also**: CB-11, CB-14, CB-17

## CB-17: Apply Filters to Transform Output

**Strength**: SHOULD

**Summary**: Chain filters with the pipe (`|`) character to transform values before output.

```html
<!-- String filters -->
{{ page.title | upcase }}
{{ page.title | downcase }}
{{ page.title | capitalize }}
{{ page.title | truncate: 50, "..." }}
{{ "hello world" | split: " " | first }}
{{ page.title | replace: " ", "-" }}

<!-- Date filter -->
{{ page.published_date | date: "%B %d, %Y" }}
<!-- Output: January 15, 2024 -->

<!-- Math filters -->
{{ collection.posts | size }}
{{ 10 | plus: 5 }}
{{ 10 | minus: 3 }}

<!-- Array filters -->
{{ page.tags | join: ", " }}
{{ page.tags | sort | join: ", " }}
{{ collection.posts | size }} posts

<!-- HTML/URL filters -->
{{ page.title | escape }}
{{ page.permalink | url_encode }}

<!-- Chaining multiple filters -->
{{ page.title | downcase | replace: " ", "-" | truncate: 30 }}
```

**Rationale**: Filters provide data transformation at the template level without requiring changes to the source data. Cobalt's Liquid supports string, math, array, date, HTML, and URL filters. Filters are chained left to right, with each filter's output becoming the next filter's input.

**See also**: CB-11, CB-14

## CB-18: Use assign and capture for Template Variables

**Strength**: CONSIDER

**Summary**: Use `{% assign %}` for simple values and `{% capture %}` for multi-line or computed content.

```html
<!-- assign: create a simple variable -->
{% assign site_name = site.title | default: "My Site" %}
<title>{{ page.title }} | {{ site_name }}</title>

<!-- capture: capture rendered output into a variable -->
{% capture post_url %}{{ site.base_url }}/{{ page.permalink }}{% endcapture %}
<link rel="canonical" href="{{ post_url }}">

<!-- capture with logic -->
{% capture page_class %}
  {% if page.is_draft %}draft{% else %}published{% endif %}
{% endcapture %}
<body class="{{ page_class | strip }}">
```

**Rationale**: `assign` creates a named variable from a value or filter expression. `capture` captures rendered template output as a string variable, useful for building complex strings or avoiding repetition. Both are scoped to the current template rendering context.

**See also**: CB-11, CB-16

## CB-19: Configure _cobalt.yml for Site-Wide Settings

**Strength**: MUST

**Summary**: Use `_cobalt.yml` to set build options, site metadata, page/post defaults, and asset settings.

```yaml
# _cobalt.yml - all settings are optional with sensible defaults

# Build options
source: "."
template_extensions:
  - md
  - liquid
include_drafts: false
ignore: []
destination: _site

# Site metadata
site:
  title: My Blog
  description: A blog about Rust
  base_url: https://example.com

# Default frontmatter for all pages and posts
default:
  excerpt_separator: "\n\n"
  is_draft: false

# Syntax highlighting
syntax_highlight:
  theme: "base16-ocean.dark"
  enabled: true

# Asset processing
assets:
  sass:
    style: Nested   # Nested | Expanded | Compact | Compressed

# Page-specific defaults
pages:
  default: {}

# Post-specific options
posts:
  title: My Blog
  description: Blog description
  dir: posts
  drafts_dir: _drafts
  order: Desc
  rss: rss.xml
  jsonfeed: ~
  publish_date_in_filename: true
  default: {}
```

**Rationale**: `_cobalt.yml` is entirely optional -- Cobalt works with built-in defaults for everything. However, configuring site title, base URL, RSS feeds, and frontmatter defaults here prevents repetition across individual content files. Frontmatter values set here are defaults that individual files can override.

**See also**: CB-08, CB-09, CB-20

## CB-20: Set Up RSS Feeds

**Strength**: SHOULD

**Summary**: Configure four required fields in `_cobalt.yml` to enable automatic RSS 2.0 feed generation.

```yaml
# _cobalt.yml - all four fields are mandatory for RSS
posts:
  title: My Blog           # Feed title
  description: A Rust blog  # Feed description
  rss: rss.xml             # Output path for the RSS file
site:
  base_url: https://example.com  # Required for absolute URLs
```

```yaml
# Each post must have title and published_date in frontmatter
---
title: "My First Post"
published_date: "2024-01-15 12:00:00 +0000"
---
```

**Rationale**: All four configuration fields (`posts.title`, `posts.description`, `posts.rss`, `site.base_url`) are mandatory per the RSS 2.0 specification -- none are optional. Posts must also provide `title` and `published_date` in their frontmatter. Cobalt also supports JSON Feed output via `posts.jsonfeed` as a separate configuration.

**See also**: CB-19, CB-06

## CB-21: Use Data Files for Shared Structured Data

**Strength**: CONSIDER

**Summary**: Place YAML, JSON, or TOML files in `_data/` to make structured data available as `site.data.<path>.<file>`.

```yaml
# _data/team/members.yml
- name: Alice
  role: Developer
- name: Bob
  role: Designer
```

```html
<!-- Access in templates via site.data path mapping -->
<ul>
{% for member in site.data.team.members %}
  <li>{{ member.name }} - {{ member.role }}</li>
{% endfor %}
</ul>
```

**Rationale**: Data files separate structured data from templates, making it reusable across multiple pages. The directory structure within `_data/` maps directly to the variable hierarchy: `_data/animals/dogs.yml` becomes `site.data.animals.dogs`. Supported formats are YAML (`.yml`/`.yaml`), JSON (`.json`), and TOML (`.toml`). Data files are distinct from frontmatter data -- data files are site-wide (`site.data`), while frontmatter data is per-page (`page.data`).

**See also**: CB-14, CB-19

## CB-22: Enable Syntax Highlighting for Code Blocks

**Strength**: SHOULD

**Summary**: Use Markdown fenced code blocks or the `{% highlight %}` Liquid tag with a configurable theme.

````markdown
<!-- Markdown method (in .md files) -->
```rust
fn main() {
    println!("Hello, world!");
}
```
````

```
<!-- Liquid tag method (in any template file) -->
{% highlight python %}
def greet(name):
    print(f"Hello, {name}!")
{% endhighlight %}
```

```yaml
# _cobalt.yml - configure theme and enable/disable
syntax_highlight:
  theme: "base16-ocean.dark"   # Default theme
  enabled: true                # Enabled by default

# Discover available options
# $ cobalt debug highlight themes
# $ cobalt debug highlight syntaxes
```

**Rationale**: Syntax highlighting is enabled by default with the `base16-ocean.dark` theme. Custom syntax definitions (Sublime Text format) can be added to `_syntaxes/`. Use `cobalt debug highlight themes` and `cobalt debug highlight syntaxes` to discover available options. Both the Markdown and Liquid methods produce the same highlighted output.

**See also**: CB-10, CB-19, CB-25

## CB-23: Configure Pagination for Post Listings

**Strength**: CONSIDER

**Summary**: Add a `pagination` block to an index page's frontmatter to split posts across multiple pages.

```yaml
# index.liquid frontmatter - basic "all posts" pagination
---
permalink: /
pagination:
  include: All              # REQUIRED: All | Tags | Categories | Dates
  per_page: 5               # Default: 5
  permalink_suffix: ./{{num}}/
  order: Desc               # Default: Desc
  sort_by: ["published_date"]
---
```

```html
<!-- Template using the paginator object -->
{% for post in paginator.pages %}
<article>
  <h2><a href="/{{ post.permalink }}">{{ post.title }}</a></h2>
  <time>{{ post.published_date | date: "%B %d, %Y" }}</time>
</article>
{% endfor %}

<!-- Navigation between pages -->
{% if paginator.previous_index %}
  <a href="/{{ paginator.previous_index_permalink }}">Newer</a>
{% endif %}
{% if paginator.next_index %}
  <a href="/{{ paginator.next_index_permalink }}">Older</a>
{% endif %}
```

```yaml
# Tag-based pagination
---
permalink: /tags
pagination:
  include: Tags
  permalink_suffix: ./{{num}}/
---
# Generates: /tags/my_tag/index.html, /tags/my_tag/2/index.html, etc.
```

**Rationale**: The `include` field is mandatory and determines the indexing mode. When pagination is active, posts are accessed through `paginator.pages`, not `collection.posts` -- they are mutually exclusive. The permalink suffix is appended after removing the file extension from the page's permalink. Forgetting to set `include` will silently prevent pagination from activating.

**See also**: CB-15, CB-19

## CB-24: Use Sass/SCSS for Stylesheets

**Strength**: CONSIDER

**Summary**: Cobalt compiles Sass/SCSS to CSS automatically during builds; configure the output style in `_cobalt.yml`.

```yaml
# _cobalt.yml
assets:
  sass:
    style: Compressed  # Nested (default) | Expanded | Compact | Compressed
```

```scss
// _sass/_variables.scss (partial - imported, not compiled standalone)
$primary: #2563eb;
$font-stack: system-ui, sans-serif;

// styles.scss (in source directory - compiled to styles.css)
@import "variables";

body {
  font-family: $font-stack;
  color: $primary;
}
```

**Rationale**: Sass support is built into Cobalt since v0.12.2 with no plugins needed. Files starting with `_` in `_sass/` are partials (imported by other Sass files, not compiled independently). The `Compressed` style produces minified CSS suitable for production. Both `.sass` (indentation syntax) and `.scss` (curly-brace syntax) are supported.

**See also**: CB-03, CB-19

**Summary**: Use `cobalt debug` subcommands to inspect configuration, file collections, and syntax highlighting options.

**Rationale**: When pages or posts are missing from the build output, `cobalt debug files <collection> --trace` explains which ignore patterns caused exclusion. `cobalt debug config` shows the effective configuration after defaults are applied, which is essential for understanding precedence between `_cobalt.yml` defaults and frontmatter overrides.

## CB-27: Use the Liquid Rust Parser and ParserBuilder

**Strength**: CONSIDER

**Summary**: Build a `Parser` via the builder pattern with `ParserBuilder::with_stdlib()` to compile Liquid templates programmatically.

```rust
// Good - standard usage: create parser with stdlib, parse, render
let parser = liquid::ParserBuilder::with_stdlib()
    .build()
    .unwrap();

let template = parser.parse("Hello, {{ name }}!").unwrap();

let globals = liquid::object!({ "name": "World" });
let output = template.render(&globals).unwrap();
assert_eq!(output, "Hello, World!");

// Good - parse from a file
let template = parser.parse_file("template.liquid").unwrap();

// Good - register custom extensions before building
let parser = liquid::ParserBuilder::with_stdlib()
    .filter(MyCustomFilter)
    .tag(MyCustomTag)
    .block(MyCustomBlock)
    .build()
    .unwrap();
```

```rust
// Bad - parser without stdlib (missing all standard tags and filters)
let parser = liquid::ParserBuilder::new()
    .build()
    .unwrap();
// template.parse("{% if x %}...{% endif %}") will FAIL
// because the "if" block is not registered
```

**Rationale**: `ParserBuilder::with_stdlib()` pre-loads all standard Liquid blocks, tags, and filters. Creating a parser with `new()` gives you an empty parser where even basic constructs like `if` and `for` are unavailable. The `Parser` is `Send + Sync` and can be shared across threads. Extensions must be registered on the `ParserBuilder` before calling `build()` -- you cannot modify a built `Parser`.

**See also**: CB-28, CB-29

## CB-28: Render Templates with the Template Object

**Strength**: CONSIDER

**Summary**: Templates are compiled once and rendered with `render(&globals)` or streamed with `render_to(&mut writer, &globals)`.

```rust
// render() returns a String
let template = liquid::ParserBuilder::with_stdlib()
    .build().unwrap()
    .parse("Liquid! {{ num | minus: 2 }}").unwrap();

let globals = liquid::object!({ "num": 4f64 });
let output = template.render(&globals).unwrap();
assert_eq!(output, "Liquid! 2");

// render_to() writes directly to a writer (avoids String allocation)
use std::io::Write;
let mut file = std::fs::File::create("output.html").unwrap();
template.render_to(&mut file, &globals).unwrap();

// Templates are Send + Sync: compile once, render concurrently
use std::sync::Arc;
let template = Arc::new(template);
// Share across threads...
```

**Rationale**: `Template` is an opaque, immutable struct produced by the parser. Its `render` method takes `&self`, so it can be called repeatedly with different globals without recompilation. Use `render_to` for large templates to avoid intermediate `String` allocation. Templates cannot be modified after compilation -- custom filters and tags must be registered on `ParserBuilder` before parsing.

**See also**: CB-27, CB-29

## CB-29: Construct Template Data with the Liquid Value System

**Strength**: CONSIDER

**Summary**: Use the `object!` macro for inline data construction, `Value` for individual values, and `to_object` for Serde conversion.

```rust
// Good - object! macro with JSON-like syntax
let globals = liquid::object!({
    "title": "My Page",
    "count": 42i64,
    "rating": 4.5f64,
    "tags": ["rust", "liquid"],
    "active": true
});

// Good - empty object
let globals = liquid::Object::new();

// Good - convert Serde-serializable types
#[derive(serde::Serialize)]
struct PageData {
    title: String,
    count: i64,
}
let data = PageData { title: "Hello".into(), count: 5 };
let obj = liquid::to_object(&data).unwrap();

// Good - manual Value construction
use liquid_core::model::Value;
let scalar = Value::scalar("hello");
let number = Value::scalar(42i64);
let nil = Value::Nil;
```

**Rationale**: The `Value` enum has four variants: `Scalar` (string/int/float/bool/datetime), `Array`, `Object`, and `Nil`. The `object!` macro provides ergonomic construction, while `to_object` bridges Serde-serializable Rust types into Liquid's data model. Note that `4` is an integer in the macro; use `4f64` for floating-point values.

**See also**: CB-27, CB-28, CB-30

## CB-30: Use ObjectView and ValueView Traits for Zero-Copy Rendering

**Strength**: CONSIDER

**Summary**: Implement `ObjectView` and `ValueView` on custom Rust structs to pass them as template data without conversion.

```rust
// Good - derive both traits for zero-copy template rendering
use liquid::ObjectView;
use liquid::ValueView;

#[derive(ObjectView, ValueView)]
struct MyGlobals {
    title: String,
    count: i64,
}

let globals = MyGlobals {
    title: "Hello".into(),
    count: 42,
};

// Template::render accepts &dyn ObjectView
let output = template.render(&globals).unwrap();

// Standard types that implement ObjectView:
// - liquid::Object (BTreeMap-based)
// - HashMap<K, V> where K: ObjectIndex, V: ValueView
// - BTreeMap<K, V> where K: ObjectIndex, V: ValueView
// - &O where O: ObjectView
```

**Rationale**: `ObjectView` and `ValueView` are borrowed-view traits that allow custom Rust types to be exposed to Liquid templates without copying into `liquid::Object`. This enables zero-copy rendering where templates borrow data from existing Rust structs. Both traits must be implemented together (`ObjectView` requires `ValueView`). The derive macros handle the implementation automatically.

**See also**: CB-29, CB-31

## CB-31: Build Sites Programmatically with cobalt::Config

**Strength**: CONSIDER

**Summary**: Use the `cobalt::Config` struct and `cobalt::build(config)` function to build sites from Rust code without the CLI.

```rust
// Good - build with defaults
let config = cobalt::Config::default();
cobalt::build(config).expect("Site build failed");

// Good - customize paths
let mut config = cobalt::Config::default();
config.source = std::path::PathBuf::from("./my-site");
config.destination = std::path::PathBuf::from("./output");
config.include_drafts = true;
cobalt::build(config).expect("Site build failed");

// Good - construct from a YAML config file
let yaml_config: cobalt_config::Config = /* load from _cobalt.yml */;
let config = cobalt::Config::from_config(yaml_config)
    .expect("Invalid configuration");
cobalt::build(config).expect("Site build failed");
```

**Rationale**: The `Config` struct has 14 public fields covering source, destination, page extensions, ignore patterns, page/post collections, site metadata, layouts path, Liquid/Markdown/syntax/asset settings, and minification. `cobalt::Config` is a re-export of `cobalt::cobalt_model::Config`; `cobalt_config::Config` is the raw deserialized YAML structure. Use `from_config` to get proper default resolution; constructing `Config` directly may miss default logic.

**See also**: CB-27, CB-30

## CB-32: Avoid Common Cobalt Pitfalls

**Strength**: AVOID

**Summary**: Avoid these frequent mistakes when working with Cobalt.

```yaml
# Bad - posts directory with underscore (wrong!)
_posts/
  my-post.md

# Good - posts directory without underscore
posts/
  my-post.md

# Bad - expecting page.content inside page body
---
title: My Page
---
This page says: {{ page.content }}  <!-- EMPTY: only works in layouts -->

# Bad - forgetting closing frontmatter delimiter
---
title: Broken
Everything below is treated as YAML, not content...

# Bad - using Markdown syntax that depends on Liquid output
<!-- Liquid runs BEFORE Markdown, so this works: -->
{{ "**bold from Liquid**" }}  <!-- Becomes <strong>bold from Liquid</strong> -->

<!-- But this does NOT work: -->
**{{ some_var }}**  <!-- The ** are processed by Markdown first? No --
                         actually Liquid runs first, so this IS fine.
                         The real issue is the reverse: Markdown cannot
                         generate Liquid expressions. -->
```

```console
# Bad - building without publishing drafts first
$ cobalt build
# Draft posts silently excluded from output

# Good - publish before building for production
$ cobalt publish posts/my-draft.md
$ cobalt build
```

**Rationale**: The most common mistakes are: (1) using `_posts/` instead of `posts/` (Cobalt differs from Jekyll here), (2) trying to access `page.content` inside a page body instead of a layout, (3) forgetting the closing `---` frontmatter delimiter, and (4) building without publishing draft posts. Understanding the Liquid-before-Markdown processing order (CB-10) also prevents template evaluation surprises.

**See also**: CB-03, CB-07, CB-08, CB-10
