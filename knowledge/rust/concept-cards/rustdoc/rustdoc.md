# Quick Definition

Rustdoc is the standard documentation generator shipped with the Rust distribution. It takes a crate root or Markdown file as input and produces HTML, CSS, and JavaScript documentation. It is most commonly invoked via `cargo doc`, which handles crate naming, output paths, and dependency resolution automatically.

# Core Definition

As stated in the source: "The standard Rust distribution ships with a tool called `rustdoc`. Its job is to generate documentation for Rust projects. On a fundamental level, Rustdoc takes as an argument either a crate root or a Markdown file, and produces HTML, CSS, and JavaScript." (Ch. 1: What is rustdoc?). Rustdoc extracts documentation from `///` (outer) and `//!` (inner) doc comments and renders them alongside the item signatures. By default, rustdoc only generates documentation for publicly reachable items; private items and public items inside private modules are excluded.

The primary way to use rustdoc is through Cargo integration. Running `cargo doc` invokes rustdoc with the correct `--crate-name`, output directory (`target/doc`), and library paths for dependencies. This means dependency documentation is also generated alongside your crate's documentation.

# Prerequisites

- None. Rustdoc is a foundational tool available in every Rust installation.

# Key Properties

1. **Automatic with Cargo**: `cargo doc` handles crate name, output path, and dependency resolution; `cargo doc --open` generates and opens in a browser
2. **Public-only by default**: Only publicly reachable items appear in output unless `--document-private-items` is passed
3. **Two input modes**: Accepts either a crate root (`.rs` file) or a standalone Markdown file
4. **Doc comment syntax**: `///` documents the next item (outer), `//!` documents the enclosing item (inner)
5. **Dependency documentation**: When using `cargo doc`, documentation for all dependencies is generated alongside the crate
6. **Crate root documentation**: Use `//!` at the crate root to document the crate itself, since nothing comes before the root module
7. **Output structure**: Generates a full website with search, navigation, and per-item pages under `doc/` (direct) or `target/doc/` (Cargo)

# Construction / Recognition

## Basic Direct Usage:
```bash
$ rustdoc src/lib.rs --crate-name mycrate
```
This produces documentation in `doc/mycrate/index.html`.

## Standard Cargo Usage:
```bash
$ cargo doc          # generate docs for crate and dependencies
$ cargo doc --open   # generate and open in browser
$ cargo doc --verbose  # see the rustdoc command Cargo executes
```
Output goes to `target/doc/`.

## Documentation Comment Patterns:
```rust
/// Documents the function below (outer doc comment)
pub fn documented_function() {}

//! Documents the enclosing module or crate (inner doc comment)
```

## Standalone Markdown:
```bash
$ rustdoc README.md
```
Produces `doc/README.html` from the Markdown contents. Cargo does not support standalone Markdown files.

# Context & Application

Rustdoc is the universal documentation tool for the Rust ecosystem. Every published crate on crates.io has its documentation generated by rustdoc and hosted on docs.rs. It is also the tool that generates the Rust standard library documentation. Understanding rustdoc's basic operation is essential for any Rust developer who writes library code, reads API documentation, or publishes crates. The tool integrates tightly with the compiler, using the same `cfg` flags and edition settings, which means generated documentation accurately reflects the code's public API.

# Examples

**Example 1** (Ch. 1): Creating a documented function:
```rust
/// foo is a function
pub fn foo() {}
```
Running `rustdoc src/lib.rs --crate-name docs` generates a page at `doc/docs/fn.foo.html` containing the text "foo is a function."

**Example 2** (Ch. 1): Demonstrating that private items are excluded:
```rust
pub fn public() {}      // documented
mod private {           // not documented
    pub fn unreachable() {} // not documented (public but inside private module)
}
```

**Example 3** (Ch. 1): How `cargo doc` invokes rustdoc internally:
```bash
$ rustdoc --crate-name docs src/lib.rs -o <path>/docs/target/doc -L dependency=<path>/docs/target/debug/deps
```
Cargo automatically provides the crate name, output directory, and dependency search paths.

**Example 4** (Ch. 1): Crate-level documentation using inner doc comments:
```rust
//! This is my first rust crate
```
When used at the crate root, this documents the crate itself.

# Common Errors

- **Error**: Running `rustdoc src/lib.rs` and finding the crate named "lib" with no documented items.
  **Correction**: Without `--crate-name`, rustdoc uses the filename. Without `pub` visibility, no items appear. Use `--crate-name mycrate` and ensure items are `pub`.

- **Error**: Expecting `rustdoc README.md` to work through Cargo.
  **Correction**: "Cargo currently does not understand standalone Markdown files, unfortunately." Use `rustdoc` directly for Markdown.

- **Error**: Writing `///` at the top of `lib.rs` expecting it to document the crate.
  **Correction**: Use `//!` (inner doc comments) at the crate root to document the crate itself. `///` documents the next item, and there is no item before the crate root.

# Common Confusions

- **Confusion**: Thinking `rustdoc` and `cargo doc` are separate tools.
  **Clarification**: `cargo doc` is a convenience wrapper that calls `rustdoc` with the correct arguments. You can see the exact invocation with `cargo doc --verbose`.

- **Confusion**: Thinking all `pub` items always appear in documentation.
  **Clarification**: A `pub` item inside a private module is not publicly reachable and will not appear unless `--document-private-items` is used. Rustdoc documents publicly *reachable* items, not merely items declared `pub`.

- **Confusion**: Thinking `///` and `//!` are interchangeable.
  **Clarification**: `///` (outer) documents the *next* item. `//!` (inner) documents the *enclosing* item. Using `///` where `//!` is needed (or vice versa) attaches documentation to the wrong item.
