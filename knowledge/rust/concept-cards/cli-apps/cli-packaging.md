# Quick Definition

Rust CLI tools can be distributed through three channels of increasing user convenience: `cargo publish` to crates.io (requires Rust to install), pre-built binary releases via CI (no Rust needed, manual install), and system package managers like Homebrew or APT (automatic updates, familiar workflow).

# Core Definition

The chapter presents three distribution strategies, ordered from easiest to set up to most convenient for end users:

1. **`cargo publish`**: Publish to crates.io. Users install with `cargo install <crate-name>`, which compiles from source into `~/.cargo/bin/`. Requires the user to have Rust and cargo installed. Best for tools targeting Rust developers. Requires a crates.io account (authorized via GitHub), a `cargo login` token, and proper `Cargo.toml` metadata (name, version, license, description, repository).

2. **Pre-built binaries**: Compile release binaries on CI (GitHub Actions, Travis CI, AppVeyor) for each target platform. Upload as GitHub release artifacts. Users download, extract, and place the binary in their PATH. No Rust installation needed. For maximum compatibility: target `x86_64-unknown-linux-musl` on Linux (static linking, no glibc dependency) and set `MACOSX_DEPLOYMENT_TARGET` on macOS. The `cross` tool (Docker-based) enables cross-compilation to exotic targets.

3. **System package managers**: Homebrew formulas, APT/AUR packages, etc. Most convenient for users (install and update via their existing package manager). Most effort to set up and maintain. Helper crates: `cargo-bundle`, `cargo-deb`, `cargo-aur`.

Additional release artifacts beyond the binary: man pages (generated by `clap_mangen`), shell completion scripts (generated by `clap_complete`), README, and license files.

# Key Properties

1. `cargo publish` uploads source to crates.io; `cargo install` compiles from source
2. crates.io requires GitHub authorization and a cargo login token
3. `Cargo.toml` metadata: name, version, authors, license, description, readme, repository, keywords, categories
4. `cargo install` places binaries in `~/.cargo/bin/` (must be in PATH)
5. `cargo build --release` produces optimized binaries in `target/release/`
6. Rust statically links dependencies by default -- binaries are self-contained
7. `x86_64-unknown-linux-musl` target produces fully static Linux binaries
8. CI builds on multiple platforms produce cross-platform release artifacts
9. The `cross` tool uses Docker containers for cross-compilation
10. `clap_mangen` generates man pages; `clap_complete` generates shell completions
11. System package managers (Homebrew, AUR, deb) provide the best user experience
12. `cargo-deb`, `cargo-aur`, `cargo-bundle` automate platform-specific packaging

# Construction / Recognition

## To Publish to crates.io:
1. Create account at crates.io (GitHub OAuth)
2. Generate API token at crates.io/me
3. Run `cargo login <token>` (one-time per machine)
4. Ensure `Cargo.toml` has required fields: name, version, license, description
5. Run `cargo publish`

## To Set Up Binary Releases:
1. Configure CI (GitHub Actions, Travis CI) to build on Linux, macOS, Windows
2. Use `cargo build --release` for each target
3. For Linux: target `x86_64-unknown-linux-musl` for static binaries
4. Package binary + README + LICENSE into archive files
5. Upload archives as GitHub release artifacts

## To Ship Additional Artifacts:
1. Generate man pages with `clap_mangen` at build time
2. Generate shell completions with `clap_complete` for bash, zsh, fish, etc.
3. Include these files in the release archive alongside the binary

## Recommended Progression:
1. Start with `cargo install` (zero setup beyond good `Cargo.toml`)
2. Add binary releases (CI builds for major platforms)
3. Finally, add system package manager support as adoption grows

# Context & Application

Distribution strategy depends on the target audience. Tools for Rust developers (like `cargo-tree` or `cargo-outdated`) work well with `cargo install` alone. Tools for general users need pre-built binaries or package manager presence. The book uses ripgrep as a model: it offers all three channels. The recommendation is to layer these approaches incrementally rather than choosing one.

**ripgrep as a distribution model**: ripgrep's README offers GitHub release downloads, package manager commands for 10+ platforms, and `cargo install ripgrep`. This layered approach maximizes accessibility.

**Key trade-off**: `cargo install` is trivial to set up but requires Rust on the user's machine and recompiles from source (slow). Pre-built binaries are fast to install but require manual updates. Package managers provide updates but require ongoing maintenance of package definitions.

# Examples

**Example 1** (Ch 8): Cargo.toml metadata for publishing:
```toml
[package]
name = "grrs"
version = "0.1.0"
authors = ["Your Name <your@email.com>"]
license = "MIT OR Apache-2.0"
description = "A tool to search files"
readme = "README.md"
homepage = "https://github.com/you/grrs"
repository = "https://github.com/you/grrs"
keywords = ["cli", "search", "demo"]
categories = ["command-line-utilities"]
```

**Example 2** (Ch 8): Installing from crates.io:
```console
$ cargo install grrs
    Downloading grrs v0.1.0
    Compiling grrs v0.1.0
    Installing ~/.cargo/bin/grrs
```

**Example 3** (Ch 8): Building a release binary:
```console
$ cargo build --release
   Compiling grrs v0.1.0
    Finished release [optimized] target(s) in 1.23s
$ ls target/release/grrs
target/release/grrs
```

**Example 4** (Ch 8): Cross-platform CI strategy (conceptual):
```yaml
# GitHub Actions matrix (conceptual)
strategy:
  matrix:
    include:
      - os: ubuntu-latest
        target: x86_64-unknown-linux-musl
      - os: macos-latest
        target: x86_64-apple-darwin
      - os: windows-latest
        target: x86_64-pc-windows-msvc
```

**Example 5** (Ch 8): Install via trust script:
```console
$ curl -LSfs https://japaric.github.io/trust/install.sh | \
    sh -s -- --git your-name/repo-name
```

# Common Errors

- **Error**: Running `cargo publish` without the `license` field in `Cargo.toml`.
  **Correction**: Add `license = "MIT OR Apache-2.0"` (or your chosen license). crates.io requires either `license` or `license-file`.

- **Error**: Distributing Linux binaries built with default glibc linking that fail on older systems.
  **Correction**: Build with `--target x86_64-unknown-linux-musl` for static binaries that work on any Linux system regardless of glibc version.

- **Error**: Forgetting that `~/.cargo/bin/` must be in the user's PATH for `cargo install` to work.
  **Correction**: The Rust installer (`rustup`) adds this to PATH. Remind users to restart their shell or source their profile after installing Rust.

# Common Confusions

- **Confusion**: Thinking `cargo publish` distributes binaries.
  **Clarification**: `cargo publish` uploads source code to crates.io. `cargo install` then compiles that source on the user's machine. No pre-built binary is involved.

- **Confusion**: Assuming cross-compilation just works with `cargo build --target`.
  **Clarification**: Cross-compilation requires the target's toolchain (linker, system libraries). The `cross` tool handles this via Docker containers. Native compilation on CI (build on each OS) is often simpler.

- **Confusion**: Thinking you must choose only one distribution channel.
  **Clarification**: The recommended approach is to layer all three: start with `cargo publish`, add binary releases, then package for system managers. ripgrep is the model example of this layered strategy.
