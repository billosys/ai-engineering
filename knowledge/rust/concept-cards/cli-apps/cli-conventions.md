# Quick Definition

CLI conventions encompass three foundational practices for well-behaved command-line applications: configuration file management (using crates like `confy`), meaningful exit codes (following BSD `sysexits.h` conventions via the `exitcode` crate), and auto-generated documentation (`--help` and man pages from `clap` definitions via `clap_mangen`).

# Core Definition

Three related conventions define a polished CLI application. **Configuration files**: The `confy` crate provides boilerplate-free config management -- define a `Serialize`/`Deserialize` struct and call `confy::load("my_app")` to get platform-appropriate config file handling automatically. **Exit codes**: Programs should emit meaningful exit codes (0 for success, specific values for specific failures). The BSD `sysexits.h` standard defines common codes, and the `exitcode` crate makes these available in Rust (e.g., `exitcode::OK`, `exitcode::CONFIG`, `exitcode::DATAERR`). Rust's default panic exit code is 101, and many tools use 1 for general failure. **Documentation rendering**: The `clap_mangen` crate auto-generates man pages from `clap` `Parser` definitions at compile time via `build.rs`, and `clap` itself generates `--help` output from the same definitions.

# Key Properties

1. `confy` handles platform-specific config file locations automatically; you provide only a `Serialize`/`Deserialize` struct and an app name
2. Exit code 0 means success; Rust uses 101 for panics; many tools use 1 for general errors
3. The BSD `sysexits.h` standard provides a common vocabulary of exit codes (available via the `exitcode` crate)
4. `exitcode` provides constants like `exitcode::OK`, `exitcode::CONFIG`, `exitcode::DATAERR` for semantic exit codes
5. Use `std::process::exit()` to set explicit exit codes in match arms for different error types
6. `clap_mangen` generates man pages from clap `Parser` struct definitions at compile time
7. Man page generation uses a `build.rs` script that calls `clap_mangen::Man::new(cmd)` and renders to a file
8. The `--help` flag is auto-generated by clap from the same `Parser` definition that drives man pages

# Construction / Recognition

## To Add Configuration File Support:
1. Add `confy` to `Cargo.toml`
2. Define a config struct with `#[derive(Debug, Serialize, Deserialize)]`
3. Call `confy::load("app_name")` to load configuration with platform-appropriate file paths

## To Add Meaningful Exit Codes:
1. Add `exitcode` to `Cargo.toml`
2. In the main function, match on result types
3. Call `std::process::exit(exitcode::OK)` for success
4. Call `std::process::exit(exitcode::CONFIG)` for configuration errors
5. Call `std::process::exit(exitcode::DATAERR)` for data errors
6. Use other `exitcode` constants as appropriate for the error type

## To Generate Man Pages:
1. Add `clap_mangen` as a build dependency
2. Define CLI args using a `clap` `Parser` struct with doc comments (these become descriptions)
3. Create a `build.rs` that constructs the `Command` from the parser and renders a man page
4. The generated `.1` file appears in the build output directory

# Context & Application

These three conventions -- config files, exit codes, and documentation -- are the "fit and finish" that distinguish a professional CLI tool from a quick script. Config files let users customize behavior without passing arguments every time. Exit codes enable shell scripts and CI pipelines to react programmatically to different failure modes (the BSD conventions are decades old and widely understood). Auto-generated man pages and `--help` output ensure documentation stays in sync with the code, since both derive from the same `clap` `Parser` definition. Together with argument parsing, error reporting, and output formatting, these conventions form the complete surface area of a well-behaved Unix CLI tool.

# Examples

**Example 1** (Ch. 10, "Using config files"): The `confy` crate makes configuration trivial: define `struct MyConfig { name: String, comfy: bool, foo: i64 }` with Serialize/Deserialize derives, then `let cfg: MyConfig = confy::load("my_app")?;` handles finding and parsing the config file on any platform.

**Example 2** (Ch. 11, "Exit codes"): Using the `exitcode` crate for semantic exit codes: match on the result and call `std::process::exit(exitcode::OK)` for success, `std::process::exit(exitcode::CONFIG)` for config-related errors like `CantReadConfig`, or `std::process::exit(exitcode::DATAERR)` for data processing errors.

**Example 3** (Ch. 14, "Rendering documentation"): A `build.rs` script generates man pages from clap definitions: `let cmd = cli::Head::command(); let man = clap_mangen::Man::new(cmd); man.render(&mut buffer)?; std::fs::write(out_dir.join("head.1"), buffer)?;`. The resulting `head.1` file can be viewed with `man`.

# Common Errors

- **Error**: Using arbitrary exit codes (e.g., always returning 1 for any error).
  **Correction**: Use the `exitcode` crate to return semantically meaningful codes from the BSD `sysexits.h` standard, so scripts and CI can distinguish between config errors, data errors, and other failure modes.

- **Error**: Writing documentation separately from clap definitions, leading to drift.
  **Correction**: Use `clap_mangen` to generate man pages from the same `Parser` struct that defines your CLI arguments, ensuring docs and code stay in sync.

# Common Confusions

- **Confusion**: Thinking exit code 1 is a universal standard for all errors.
  **Clarification**: While many tools use 1 for general failure, the BSD `sysexits.h` defines specific codes for different error categories (e.g., 64 for usage errors, 78 for config errors). The `exitcode` crate makes these available as named constants.

- **Confusion**: Thinking config file location must be manually specified per platform.
  **Clarification**: The `confy` crate automatically determines the platform-appropriate config file location. You provide only the app name and config struct.
