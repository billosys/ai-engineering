# Quick Definition

Rust's debug info pipeline spans from MIR through LLVM to the debugger. Rustc communicates type, source, and symbol information to LLVM via the DIBuilder API, LLVM translates this to DWARF (for `*-gnu` targets) or PDB/CodeView (for `*-msvc` targets), and debuggers (GDB, LLDB, CDB) interpret the result. Because Rust "pretends" to be C/C++ in its debug info output, debugger visualizer scripts (Python for GDB/LLDB, Natvis for CDB) are essential for presenting Rust types in a meaningful way.

# Core Definition

"Debug info is a collection of information generated by the compiler that allows debuggers to correctly interpret the state of a program while it is running. That includes things like mapping instruction addresses to lines of code in the source file, and type layout information so that bytes in memory can be read and displayed in a meaningful way." (Ch. 24, introduction)

The debug info stack has six layers: (1) rustc inspects MIR and communicates to LLVM, (2) LLVM translates to target-specific format, (3) debugger reads and interprets the debug info, (4) built-in debugger formatting, (5) user-defined visualizer scripts, (6) debugger frontend display. (Ch. 24, overview)

"It is important to keep in mind that the goal is not necessarily 'represent types exactly how they appear in Rust', rather it is to represent them in a way that allows debuggers to most accurately reconstruct the data during debugging." (Ch. 24, Type Information)

# Key Properties

1. **Three debug info formats**: DWARF for `*-gnu` targets (bundled in binary or separate), PDB/CodeView for `*-msvc` targets (separate `.pdb` file); CodeView has C-era origins with limited feature set
2. **Three supported debuggers**: GDB (DWARF, full Rust support, Python scripts), LLDB (DWARF and PDB, partial Rust support, Python scripts), CDB (PDB only, no native Rust support, Natvis scripts)
3. **C/C++ pretense**: "Rust's generated DI nodes 'pretend' to be C/C++ for both CDB and LLDB's sake" -- LLDB uses the Clang TypeSystem plugin, imposing C++ semantics on Rust types
4. **Pointer/reference DI**: Wide pointers are structs with `data_ptr` and `length`; non-wide pointers/references/Box are plain pointer nodes with no `mut`/non-`mut` distinction; C++ references have different semantics than Rust references
5. **MSVC type name transformations**: `&str` becomes `ref$<str$>`, `[T; N]` becomes `array$<T, N>`, `RustEnum` becomes `enum2$<RustEnum>`, `usize` becomes `size_t`, consecutive `>` must be separated by space to avoid parser confusion
6. **Enum representation**: DWARF uses `DW_TAG_variant_part` with discriminant member and variant nodes; PDB uses a C-equivalent discriminated union with `VariantNames` enum, per-variant structs, static `NAME` and `DISCR_EXACT` fields, and a `tag` field
7. **LLVM debug records**: Tags are always stored as DWARF tags internally; if the target needs PDB, LLVM translates DWARF tags to CodeView counterparts during codegen via `CodeViewDebug.cpp`
8. **LLDB TypeSystem architecture**: Plugin pipeline of `ASTParser` -> `TypeSystem` -> `ExpressionParser`/`Language`; Rust currently uses `TypeSystemClang` (C++ types), not a dedicated `TypeSystemRust`
9. **Synthetic providers**: Python classes wrapping `SBValue` with methods: `__init__`, `update` (caching control), `has_children`, `num_children`, `get_child_index`, `get_child_at_index`, `get_type_name`, `get_value`
10. **`update` caching semantics**: Return `True` to reuse cached children (same addresses and count), `False` to flush the cache; only parent-to-child relationship matters; pointer-based cache means overwritten values at same address auto-update
11. **`#[debugger_visualizer]` attribute**: Embeds GDB Python scripts in `.debug_gdb_scripts` section; Natvis files via `/NATVIS` linker option into PDB; LLDB not yet supported (formatter bytecode is planned)
12. **Performance concerns**: Every millisecond in visualizer scripts delays user output; Python-specific tips include using tuples over lists, avoiding regex, caching with `__slots__`, preferring local variable access, using lazy computation

# Construction / Recognition

## To Write an LLDB Synthetic Provider:
1. Define a class with `__init__(self, valobj, _dict)` storing `valobj`
2. Implement `update()` to store/refresh mutable state (heap pointer, length, etc.); return `True` if children are unchanged, `False` otherwise
3. Implement `num_children()` returning the number of displayed children
4. Implement `get_child_index(name)` mapping names to indices (use `UINT32_MAX - N` for hidden fields)
5. Implement `get_child_at_index(index)` returning `SBValue` children via `CreateValueFromAddress`, `Clone`, etc.
6. Optionally implement `get_type_name()` for display name and `get_value()` for primitive representation
7. Register with `type synthetic add -l module.ClassName -x "regex" --category Rust`

## To Register Providers via __lldb_init_module:
1. Create the `Rust` category via `debugger.GetCategory("Rust")` or `debugger.CreateCategory("Rust")`
2. Enable it with `rust_cat.SetEnabled(True)`
3. Create `SBTypeNameSpecifier`, `SBTypeSynthetic`, and `SBTypeSummary` objects
4. Add to category with `rust_cat.AddTypeSynthetic()` and `rust_cat.AddSummary()`

## To Handle DWARF vs PDB Differences:
1. Gate format-specific logic with `cpp_like_debuginfo` checks in the compiler
2. In visualizer scripts, use `SBType.GetTemplateArgumentType()` for DWARF, fall back to name parsing for PDB
3. Account for MSVC name transformations (`ref$<str$>` vs `&str`, etc.)

# Context & Application

This is the largest chapter in the compiler guide at 1,801 lines, reflecting the enormous surface area of debug info generation. The chapter effectively spans three domains: compiler internals (how rustc generates debug info via DIBuilder), debug info format details (DWARF and PDB/CodeView specifications and limitations), and debugger internals (how GDB, LLDB, and CDB consume and present the information).

The most important architectural insight is that Rust currently does not have its own TypeSystem plugin in LLDB. Instead, it rides on `TypeSystemClang`, which imposes C++ semantics. This is why debug info must "pretend" to be C/C++, leading to various workarounds like the enum representation as a C union in PDB, and the inability to distinguish `&T` from `*const T` in debug output. A `TypeSystemRust` would resolve many of these issues but is a massive undertaking.

The visualizer script section is extremely detailed and practical, including a complete worked example of a `Vec<T>` synthetic provider that demonstrates hidden fields (length and capacity accessible via name but not displayed), the `update` caching protocol, and type name normalization. The performance section is unusually practical for a compiler guide, reflecting real-world pain points with slow debugger output on large stack frames.

The `#[debugger_visualizer]` attribute enables library authors to embed pretty printers directly in their crate, providing a seamless debugging experience for downstream users without requiring manual debugger configuration.

# Examples

**Example 1** (Ch. 24, Enum in DWARF): Discriminated union hierarchy:
```txt
DW_TAG_structure_type      (top-level type)
  DW_TAG_variant_part      (variant part)
    DW_AT_discr            (reference to discriminant DW_TAG_member)
    DW_TAG_member          (discriminant member)
    DW_TAG_variant         (variant 1)
    DW_TAG_variant         (variant 2)
  DW_TAG_structure_type    (type of variant 1)
  DW_TAG_structure_type    (type of variant 2)
```

**Example 2** (Ch. 24, Vec synthetic provider -- get_child_at_index):
```python
def get_child_at_index(self, index: int) -> SBValue:
    if index == UINT32_MAX - 1:
        return self.valobj.GetChildMemberWithName("len")
    if index == UINT32_MAX - 2:
        return (
            self.valobj.GetChildMemberWithName("buf")
            .GetChildMemberWithName("inner")
            .GetChildMemberWithName("cap")
            .GetChildAtIndex(0)
            .Clone("capacity")
        )
    addr = self.data_ptr.GetValueAsAddress()
    addr += index * self.element_type.GetByteSize()
    return self.valobj.CreateValueFromAddress(
        f"[{index}]", addr, self.element_type
    )
```

**Example 3** (Ch. 24, DWARF extension for vtables): Rust emits a custom `DW_AT_containing_type` on vtable structures:
```asm
<1><1a9>: Abbrev Number: 3 (DW_TAG_structure_type)
   <1aa>   DW_AT_containing_type: <0x1b4>
   <1ae>   DW_AT_name        : vtable
   <1b2>   DW_AT_byte_size   : 0
   <1b3>   DW_AT_alignment   : 8
```
This lets debuggers dissect trait object pointers to find the concrete payload type.

# Common Errors

- **Error**: Using C++ `reference` DI nodes for Rust references.
  **Correction**: C++ references "do not necessarily occupy storage" and "there are no arrays of references, no pointers to references, and no references to references." Rust references have different semantics; use pointer nodes instead (with potential typedef wrappers).

- **Error**: Writing visualizer scripts that use regex for string manipulation.
  **Correction**: "Regexes are slow and should be avoided when simple string manipulation will do." Use f-strings and basic string operations for hot paths.

- **Error**: Returning `True` from `update()` when the number of children or their addresses have changed.
  **Correction**: "Returning True in the wrong circumstances will result in the debugger outputting incorrect information." Return `True` only when `data_ptr` address and `length` are unchanged.

# Common Confusions

- **Confusion**: Thinking `num_children` must return the total number of fields in the type.
  **Clarification**: "Fields can be intentionally 'hidden' from LLDB while still being accessible to the user." For `Vec`, `num_children` returns only the element count; `len` and `capacity` are accessible by name via `get_child_index` at high index values.

- **Confusion**: Thinking LLDB stores debug info tags in the target format.
  **Clarification**: "Tags within the debug records are always stored as DWARF tags. If the target calls for PDB debug info, during codegen the debug records will then be passed through a module that translates the DWARF tags to their CodeView counterparts."

- **Confusion**: Thinking a `TypeSystemRust` plugin is needed for basic Rust debugging in LLDB.
  **Clarification**: Rust currently uses `TypeSystemClang` (C++ types) with Python visualizer scripts for pretty-printing. A dedicated TypeSystem would improve the experience but is not required for basic debugging.
