# Quick Definition

Bootstrapping is the process of using an older compiler to compile a newer version of the same compiler. Rust's build system (`./x`) downloads the current beta release as the stage0 compiler, uses it to compile the source code into the stage1 compiler, and then optionally uses stage1 to compile itself into stage2 (the truly current compiler). Stage1 is sufficient for most development; stage2 is what ships with `rustup` and is needed for distribution and `rustc_private`-dependent tools. The build system is organized around `Step` traits that form a dependency graph, managed by `Builder::ensure` for deduplication.

# Core Definition

Rustc was originally written in OCaml but has been self-hosting for many years. The only way to build a modern rustc is with a slightly older version. The `./x` build tool automates this by downloading the current beta release as the starting point.

The bootstrap process has four stages:

**Stage 0**: The pre-compiled beta compiler and its standard library, downloaded automatically. It is used only to compile the bootstrap tool itself and the compiler source code.

**Stage 1**: The compiler built from current source code, compiled by the stage0 compiler. "The `stage1/bin/rustc` is a fully functional compiler built with stage0 (precompiled) compiler and std." For most development, `--stage 1` is sufficient.

**Stage 2**: The compiler rebuilt using the stage1 compiler with the in-tree standard library. This is important because "the ABI generated by the `stage0` compiler may not match the ABI that would have been made by the `stage1` compiler, which can cause problems for dynamic libraries, tests and tools using `rustc_private`." The stage2 compiler is what ships via `rustup`.

**Stage 3**: An optional verification step. The libraries are rebuilt with the stage2 compiler and the result should be identical to stage2, confirming the compiler is deterministic.

Two mechanisms resolve the chicken-and-egg problem of using nightly features in a beta-compiled compiler: `cfg(bootstrap)` / `cfg(not(bootstrap))` for recently stabilized features that need a feature gate only when built by stage0, and `RUSTC_BOOTSTRAP=1` which allows `#![feature(...)]` on a non-nightly compiler (used exclusively for bootstrapping).

The build system's core abstraction is the `Step` trait. Each `Step` represents a granular build action (like a Makefile rule). Steps are chained via `Builder::ensure`, which executes a step only if it hasn't already been run. Steps define `run` (the work), `should_run` (CLI matching), and `make_run` (direct CLI invocations only).

# Key Properties

1. **Stage convention**: `--stage N` means running the stage N compiler; a "stage N artifact" is produced by the stage N compiler; the stage N+1 compiler is assembled from stage N artifacts (called "uplifting")
2. **Two std libraries**: The stage N compiler is *linked to* stage N-1 std but *uses* stage N std for compiling programs -- these may differ in ABI
3. **Default stages per command**: `check`/`clippy`/`doc`/`build`/`test` default to stage 1; `dist`/`install`/`bench` default to stage 2
4. **`--keep-stage`**: Skips rebuilding std when ABI hasn't changed, significantly speeding up iteration; safe when only compiler logic (not data layout) has changed
5. **Sysroot contents**: Libraries (`libstd`, `libtest`, `libproc_macro`), compiler crates (via `rustc_private`), and `libLLVM.so`; accessible via `rustc --print sysroot`
6. **`-Z force-unstable-if-unmarked`**: Applied to all compiler and std crates to mark them as requiring `#![feature(rustc_private)]` for external access
7. **Cross-compiling**: When HOST != TARGET, stage2 must recompile std for the target (cannot reuse stage1 std which only runs on the host)
8. **Bootstrap flag passing**: `RUSTFLAGS_BOOTSTRAP` for stage0, `RUSTFLAGS_NOT_BOOTSTRAP` for later stages, `MAGIC_EXTRA_RUSTFLAGS` to bypass cargo cache
9. **Three tool modes**: `Mode::ToolBootstrap` (stage0 only, no compiler deps), `Mode::ToolStd` (locally-built std), `Mode::ToolRustcPrivate` (depends on compiler internals)
10. **Bootstrap tracing**: `BOOTSTRAP_TRACING=trace` enables structured logging, Chrome trace files, GraphViz step graphs, and command execution summaries

# Construction / Recognition

## Entry Points for the Build System:

1. Shell script `./x` or `./x.ps1` or `make`
2. Python wrapper `x.py`
3. `src/bootstrap/bootstrap.py`
4. `src/bootstrap/src/bin/main.rs`

## Common Build Commands:

```console
./x build library       # Build the standard library (stage1)
./x test tests/ui       # Build stage1 compiler, run UI tests
./x test --stage 0 library/std  # Test std without building rustc from source
./x build --stage 2     # Full two-stage build (for distribution)
./x fmt                 # Format code with pinned rustfmt
```

## What NOT to Do:

- `./x test --stage 0 tests/ui` -- runs tests on the beta compiler, not your changes
- `./x build --stage 0 compiler/rustc` -- builds compiler without libstd or libcore

## cfg(bootstrap) Update Dance (for cyclic dependencies):

1. Accept both old and new behavior in the compiler
2. Use `#[cfg(bootstrap)]` for old behavior and `#[cfg(not(bootstrap))]` for new in the external crate
3. Update the crate version in the compiler
4. Remove the old behavior from the compiler

# Context & Application

- **Day-to-day compiler development**: Use `--stage 1` for building and testing; use `--keep-stage 1` when only changing compiler logic (not std)
- **Distribution builds**: Use `--stage 2` (the default for `dist` and `install`) to produce ABI-correct artifacts
- **Cross-compilation**: When building for a different architecture, stage2 must recompile std for the target
- **Tool development**: Choose the appropriate `Mode` (ToolBootstrap, ToolStd, ToolRustcPrivate) based on dependencies
- **Performance debugging**: Use `BOOTSTRAP_TRACING` to generate Chrome traces and step dependency graphs

# Examples

**Example 1**: The bootstrap stage flow (simplified):
```
stage0 compiler (beta) --[compiles]--> stage0 artifacts --[copy]--> stage1 compiler
stage1 compiler --[compiles]--> stage1 artifacts --[copy]--> stage2 compiler
```

**Example 2**: Using `cfg(bootstrap)` for a recently stabilized feature:
```rust
#[cfg(bootstrap)]
#![feature(some_newly_stable_feature)]  // needed when built by beta

#[cfg(not(bootstrap))]
// feature is already stable in stage1+
```

**Example 3**: Viewing what's in the sysroot:
```console
$ ls $(rustc --print sysroot)/lib
libLLVM-11-rust-1.50.0-nightly.so    librustc_driver-4f0cc9f50e53f0ba.so
libstd-25c6acf8063a3802.so           libtest-57470d2aa8f7aa83.so
```

**Example 4**: Enabling bootstrap tracing with filtered output:
```bash
BOOTSTRAP_TRACING=STEP=info,COMMAND=trace ./x build library --stage 1
```

# Common Errors

- **Error**: Using `RUSTC_BOOTSTRAP=1` in non-bootstrapping contexts.
  **Correction**: "Setting `RUSTC_BOOTSTRAP=1` should never be used except when bootstrapping the compiler." It breaks stability guarantees by allowing `#![feature(...)]` on stable/beta compilers.

- **Error**: Benchmarking or testing with a stage1 compiler when ABI correctness matters.
  **Correction**: Stage1 was compiled by stage0 (beta), so its ABI may not match the in-tree compiler's expectations. Use stage2 for distribution, `rustc_private` tools, and dynamic library testing.

- **Error**: Running `./x test --stage 0 tests/ui` expecting to test your changes.
  **Correction**: Stage 0 tests run against the beta compiler. Use `./x test tests/ui` (which defaults to stage 1) to test your source changes.

# Common Confusions

- **Confusion**: Stage1 and stage2 compilers produce different results.
  **Clarification**: In principle, stage1 and stage2 should produce identical code. The difference is what they were *compiled by*. Stage1 was compiled by beta (stage0), so it may have different ABI characteristics. Stage2 was compiled by stage1 (your source), making it the "truly current" compiler.

- **Confusion**: `--keep-stage` is always safe to use.
  **Clarification**: `--keep-stage 1` assumes the ABI hasn't changed between compiles. "If the ABI hasn't changed, you're good to go." But if you've changed data layouts, memory representations, or MIR structure, you must do a full rebuild.

- **Confusion**: The sysroot's `lib/` and `lib/rustlib/` directories serve the same purpose.
  **Clarification**: `lib/` contains compiler runtime dependencies (libstd, librustc_driver). `lib/rustlib/` contains internal implementation libraries (hashbrown, cfg_if) and is part of the linker search path. `lib/` is never part of the linker search path.
