# Quick Definition

A self-referential type is a struct that contains a reference (or pointer) from one of its fields to another of its fields. Such types are fundamentally broken by Rust's move semantics because internal references are invisible to the compiler and become dangling when the struct is moved.

# Core Definition

"The compiler only knows about references from outside an object into the object (such as the above example, or a reference to a field of an object). A reference entirely within an object would be invisible to the compiler." The source illustrates this with a hypothetical struct `Bad` where `b.r` points to `b.field`. "In `let a = b;`, the internal reference `b.r` to `b.field` is invisible to the compiler, so it looks like there are no references to `b` and therefore the move to `a` would be ok. However if that happened, then after the move, `a.r` would not point to `a.field` as we'd like, but to invalid memory at the old location of `b.field`, violating Rust's safety guarantees." (Async Reference, Ch. 2: Pinning, "Move semantics" section)

# Prerequisites

- **Place** — Self-referential types have fields that point to the places of other fields within the same object
- **Move semantics** — The fundamental problem with self-referential types is that moves change the address of the object, invalidating internal references

# Key Properties

1. A self-referential type contains a pointer/reference from one field to another field of the same struct instance
2. Internal references are invisible to the Rust borrow checker — the compiler cannot detect them
3. Moving a self-referential type invalidates the internal reference, because the reference still points to the old memory location
4. Rust does not have a `'self` lifetime — the hypothetical `&'self T` syntax shown in the source does not exist
5. Self-referential types are required for async function implementations, where future state machines store variables as fields that may reference each other
6. Pinning is the mechanism Rust uses to make self-referential types safe (by guaranteeing the object won't move)

# Construction / Recognition

## To Recognize a Self-Referential Type

1. Look for a struct where one field stores a pointer or reference to another field of the same struct
2. In async Rust, futures generated by async functions are self-referential when local variables reference each other across await points
3. Look for types that are `!Unpin` — this often indicates address sensitivity caused by self-references

## Why Self-Referential Types Cannot Be Written Directly in Safe Rust

1. Rust has no `'self` lifetime to express that a reference points within the same struct
2. The borrow checker cannot track internal references
3. Without pinning, moving the struct would silently create dangling pointers
4. The only safe way to have self-referential types is through pinning (which guarantees the object won't move) combined with unsafe code to set up the internal references

# Context & Application

Self-referential types are the primary motivation for pinning in Rust. Async functions compile into state machines (futures) where local variables become struct fields. When those variables reference each other, the resulting future is self-referential. Without pinning, moving such a future would invalidate its internal references. This is why `Future::poll` takes `Pin<&mut Self>` — it guarantees the future will not be moved while it contains self-references.

**Typical contexts:**

- Async function state machines (the compiler-generated futures)
- Any data structure that needs internal pointers (linked lists, intrusive collections)
- Understanding why `Pin` exists and why `Future::poll` requires a pinned self

**Important note from source:** "pinning is a low-level building block designed specifically for the implementation of async Rust... it is not an out-of-the-box solution for self-referential fields." (Ch. 2, footnote)

# Examples

**Example 1** (Ch. 2, "Move semantics" section): The hypothetical `Bad` struct:

```rust,norun
struct Bad {
    field: u64,
    r: &'self u64,
}
```

If `b` is an instance where `b.r` points to `b.field`, then `let a = b;` would move the struct but `a.r` would still point to the old location of `b.field` — now invalid memory.

**Example 2** (Ch. 2, "TL;DR" section): Async function futures are self-referential: "async functions ... are implemented as data structures where variables are stored as fields, since variables may reference each other, fields of a future implementing an async function must be able to reference each other."

# Common Errors

- **Error**: Attempting to create a self-referential struct using standard Rust references with a `'self` lifetime.
  **Correction**: Rust has no `'self` lifetime. Self-referential types require either unsafe code with raw pointers (and pinning for safety) or crate-level abstractions like `ouroboros` or `self_cell`.

# Common Confusions

- **Confusion**: Believing the compiler will automatically adjust internal references when a struct is moved.
  **Clarification**: The compiler has no knowledge of references internal to an object. Internal references store absolute memory addresses, not relative offsets. They are not updated during a move.

- **Confusion**: Thinking pinning is a general-purpose solution for creating arbitrary self-referential structs.
  **Clarification**: Per the source's footnote, pinning "was not designed to be a general-purpose mechanism, and in particular is not an out-of-the-box solution for self-referential fields." Using pinning for non-async self-referential types requires extensive unsafe code.
