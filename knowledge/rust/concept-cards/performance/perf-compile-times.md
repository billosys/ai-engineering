# Quick Definition

Rust compile times can be reduced through code-level changes: visualize the crate dependency graph with `cargo build --timings` to find serialization bottlenecks; identify expensive macros with `-Zmacro-stats` and `cargo-expand`; diagnose LLVM IR bloat from generic function monomorphization with `cargo llvm-lines`; and reduce IR by shrinking generic functions, extracting non-generic inner functions, or replacing generic utility calls like `Option::map` with `match`.

# Core Definition

Chapter 18 of The Rust Performance Book addresses compile-time optimization through code modifications (as distinct from build configuration changes covered earlier in the book). It covers four techniques. **Visualization**: `cargo build --timings` produces an HTML Gantt chart showing crate-level compilation dependencies, revealing parallelism and identifying large crates that serialize the build. **Macros**: The `-Zmacro-stats` nightly flag reports the amount of code generated by procedural and declarative macros; `cargo-expand` shows the generated code itself. If a macro generates code comparable in volume to hand-written code, it may be worth removing or replacing with a cheaper alternative, or modifying the macro to generate less code. **LLVM IR**: LLVM's backend execution is often a large part of compile time, especially when the Rust frontend generates excessive IR. `cargo llvm-lines` identifies which Rust functions produce the most LLVM IR; generic functions are the usual culprits because they are instantiated (monomorphized) for each concrete type -- potentially dozens or hundreds of times in large programs. **Fixing IR bloat**: (a) make the generic function smaller; (b) extract non-generic portions into a non-generic inner function that is instantiated only once (the `std::fs::read` pattern); (c) replace common generic utility calls like `Option::map` and `Result::map_err` with equivalent `match` expressions to avoid their monomorphization overhead. These changes can also reduce binary size. (Ch. 18: Compile Times)

# Key Properties

1. `cargo build --timings` produces an HTML Gantt chart showing crate-level build dependencies and parallelism
2. Large crates that serialize compilation may benefit from being split into smaller crates to increase build parallelism
3. `-Zmacro-stats` (nightly) reports code volume generated by both procedural and declarative macros; procedural macros are usually more notable
4. `cargo-expand` shows the full expanded output of macros, useful for understanding what a macro generates
5. Macros generating code volume comparable to hand-written code are candidates for removal, replacement, or modification to generate less
6. LLVM backend execution is often a large fraction of compile time, driven by the volume of IR the Rust frontend generates
7. `cargo llvm-lines` identifies which functions generate the most LLVM IR
8. Generic functions are the primary source of IR bloat because each concrete type instantiation (monomorphization) produces a separate copy of the function's IR
9. Making a generic function smaller directly reduces the IR produced by each instantiation
10. The "non-generic inner function" pattern extracts the type-independent body into an inner function that is instantiated only once, with the generic outer function serving as a thin conversion wrapper (exemplified by `std::fs::read`)
11. Replacing calls to generic utility methods like `Option::map` and `Result::map_err` with equivalent `match` expressions avoids their monomorphization overhead
12. These compile-time optimizations can also reduce binary size as a side benefit

# Construction / Recognition

## To Diagnose Compile Time Bottlenecks:
1. Run `cargo build --timings` and open the generated HTML file
2. Examine the Gantt chart for crates that serialize the build (long bars with no parallel work)
3. Consider splitting large serializing crates into smaller ones

## To Diagnose Macro Overhead:
1. Run `cargo +nightly rustc -- -Zmacro-stats` for a single crate, or `RUSTFLAGS="-Zmacro-stats" cargo +nightly build` for all crates
2. Identify macros generating disproportionate code volume
3. Use `cargo-expand` to see the actual generated code
4. Consider removing, replacing, or modifying the macro to reduce output

## To Diagnose and Fix LLVM IR Bloat:
1. Run `cargo llvm-lines` to identify functions generating the most IR
2. For each bloated generic function, consider:
   a. Making the function body smaller
   b. Extracting non-generic logic into a non-generic inner function
   c. Replacing generic utility calls (`Option::map`, `Result::map_err`) with `match`

## The Non-Generic Inner Function Pattern:
1. Write the generic outer function accepting the generic parameter (e.g., `P: AsRef<Path>`)
2. Define an inner `fn` that takes the concrete type (e.g., `&Path`)
3. The outer function converts the generic argument and delegates to the inner function
4. The inner function contains all the actual logic and is instantiated only once

# Context & Application

Compile times are a well-known pain point for Rust developers, and this chapter bridges the gap between build configuration (covered earlier in the book) and code-level changes. The LLVM IR bloat problem is particularly insidious because it is invisible without tooling -- a seemingly simple generic function can generate enormous amounts of IR when instantiated for many types across a large codebase. The `std::fs::read` pattern of extracting a non-generic inner function is an elegant solution that has been widely adopted in the Rust standard library and ecosystem.

The macro overhead concern reflects the increasing use of procedural macros (derive macros, attribute macros) in the Rust ecosystem. While convenient, each macro invocation can generate substantial code that the compiler must then process. The `-Zmacro-stats` flag (nightly-only) was specifically developed to make this cost visible.

The external resource "Tips for Faster Rust Compile Times" by Corrode is referenced for additional techniques beyond those covered in this chapter.

# Examples

**Example 1** (Ch. 18, "LLVM IR"): The `std::fs::read` function demonstrates the non-generic inner function pattern:
```rust
pub fn read<P: AsRef<Path>>(path: P) -> io::Result<Vec<u8>> {
    fn inner(path: &Path) -> io::Result<Vec<u8>> {
        let mut file = File::open(path)?;
        let size = file.metadata().map(|m| m.len()).unwrap_or(0);
        let mut bytes = Vec::with_capacity(size as usize);
        io::default_read_to_end(&mut file, &mut bytes)?;
        Ok(bytes)
    }
    inner(path.as_ref())
}
```
The generic outer function only does the `AsRef` conversion; the non-generic `inner` function contains all the logic and is instantiated once regardless of how many types `P` is.

**Example 2** (Ch. 18, "Macros"): The Bevy game engine (issue #19873) investigated macro-generated code volume. The `derive(Arbitrary)` macro was found to generate excessive code in fuzz testing contexts.

**Example 3** (Ch. 18, "LLVM IR"): The Servo browser engine (issue #26585) saw large compile time improvements from reducing LLVM IR generated by generic functions, demonstrating that the effects are occasionally very large.

**Example 4** (Ch. 18, "LLVM IR"): Replacing `Option::map` and `Result::map_err` calls with equivalent `match` expressions in hot generic code paths reduced monomorphization overhead because these utility methods are themselves generic and generate IR for each instantiation.

# Common Errors

- **Error**: Writing large generic function bodies that are instantiated for many types, causing LLVM IR bloat.
  **Correction**: Extract the non-generic logic into a non-generic inner function. The generic outer function should only perform type conversion and delegate.

- **Error**: Assuming macro-generated code is "free" because it is not hand-written.
  **Correction**: Use `-Zmacro-stats` to measure macro code generation. If a macro generates code volume comparable to hand-written code, consider removing, replacing, or modifying it.

- **Error**: Not checking the build dependency graph for serialization bottlenecks.
  **Correction**: Run `cargo build --timings` to visualize the Gantt chart. Large crates that block parallelism may benefit from being split.

# Common Confusions

- **Confusion**: Thinking compile time optimization only involves build configuration (linker, codegen units, opt-level).
  **Clarification**: Code-level changes -- reducing generic function size, managing macro output, improving crate parallelism -- can be equally or more impactful.

- **Confusion**: Believing `Option::map` and `Result::map_err` are zero-cost abstractions with respect to compile time.
  **Clarification**: While they are zero-cost at runtime, they are generic functions that generate IR for each instantiation. In heavily generic code, replacing them with `match` can measurably reduce compile times.

- **Confusion**: Thinking binary size and compile time are unrelated concerns.
  **Clarification**: LLVM IR reduction improves both compile time (less IR for LLVM to process) and binary size (fewer monomorphized function copies in the output).
