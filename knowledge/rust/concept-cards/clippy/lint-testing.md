# Quick Definition

Clippy lints are tested through UI tests -- plain Rust files whose compiler output is compared against `.stderr` reference files. The `cargo uitest` command runs tests, `cargo bless` generates/updates reference files, and `lintcheck` validates lints against a set of real-world crates. The process is test-driven: tests are written before lint logic.

# Core Definition

The source frames Clippy development as a "Test-Driven Development (TDD) process because our first task before implementing any logic for a new lint is to write some test cases."

**UI tests** are the primary testing mechanism. Each test is a plain Rust file in `tests/ui/` that contains code the lint should check. Clippy's output is compared against a `.stderr` file. The test file must include error marker annotations (`//~^ lint_name`) at lines where lint warnings are expected. At least one error marker must be present (unless the file uses `//@check-pass`).

**Test workflow**:
1. Write test cases in `tests/ui/<lint_name>.rs` with `//~^ lint_name` annotations
2. Run `TESTNAME=<name> cargo uitest` to execute the test (will fail initially with TDD)
3. Implement lint logic iteratively, re-running `cargo uitest` to check progress
4. When satisfied with output, run `TESTNAME=<name> cargo uibless` to generate `.stderr` files
5. Run `cargo test` for the full suite (including dogfood tests)
6. Commit the `.stderr` (and `.fixed` if applicable) files with the lint code

**Specialized test types**:
- **Cargo lints**: Tested in `tests/ui-cargo/` with fail/pass `Cargo.toml` manifests and minimal crates
- **TOML tests**: Configuration tested in `tests/ui-toml/` with `clippy.toml` + test file pairs
- **Rustfix tests**: Auto-generated `.fixed` files from `span_lint_and_sugg` suggestions
- **Dogfood tests**: Ensure Clippy does not lint its own codebase (`cargo dev dogfood`)
- **Lintcheck**: Run Clippy on real-world crates to validate against false positives

# Key Properties

1. UI tests compare Clippy output against `.stderr` reference files
2. Test files are plain Rust files in `tests/ui/` with `#![warn(clippy::<lint_name>)]` at the top
3. Error annotations use `//~^ lint_name` on the line below the expected warning
4. `cargo uitest` runs only UI tests; `cargo test` runs the full suite
5. `cargo bless` (or `cargo uibless`) generates/updates `.stderr` and `.fixed` files
6. `TESTNAME=<name>` environment variable filters to specific test files
7. Multiple tests can be run with comma-separated names: `TESTNAME=foo,bar,baz`
8. `.stderr` and `.fixed` files must be committed alongside lint code
9. Empty `.stderr` and `.fixed` files should be removed, not committed
10. Only commit files changed by `cargo bless` for your specific lint
11. Cargo lints use `tests/ui-cargo/<lint>/fail/` and `tests/ui-cargo/<lint>/pass/` directory structure
12. Configuration tests use `tests/ui-toml/` with a `clippy.toml` file in the test directory
13. `//@no-rustfix` suppresses automatic `.fixed` file generation for suggestions that may produce invalid code

# Construction / Recognition

## To Write Test Cases for a New Lint:
1. Open the generated `tests/ui/<lint_name>.rs` file
2. Add `#![warn(clippy::<lint_name>)]` at the top to enable the lint
3. Write code examples that should trigger the lint
4. Add `//~^ lint_name` annotations on the line after each expected warning
5. Write code examples that should NOT trigger the lint (to catch false positives)
6. Include edge cases: method calls, trait implementations, closures, etc.

## To Run and Bless Tests:
1. During development: `TESTNAME=<name> cargo uitest` (iterative, will fail until implementation matches expectations)
2. To generate reference files: `TESTNAME=<name> cargo uibless`
3. To verify the full suite: `cargo test`
4. To run dogfood: `cargo dev dogfood`
5. To validate against real crates: `cargo lintcheck`

## To Test Cargo Lints:
1. Run `cargo dev new_lint --name=<name> --pass=late --category=cargo` to generate the test structure
2. Edit `tests/ui-cargo/<name>/fail/Cargo.toml` with content that should trigger the lint
3. Edit `tests/ui-cargo/<name>/pass/Cargo.toml` with content that should not trigger
4. Add additional cases by copying and renaming directories under `tests/ui-cargo/<name>/`

## To Test Configuration:
1. Create a subdirectory in `tests/ui-toml/` with a fitting name
2. Add a `clippy.toml` file with the configuration value
3. Add a `.rs` test file that should exhibit different behavior based on the configuration
4. Run `cargo bless` to generate `.stderr` files

# Context & Application

The TDD approach is intentional and practical. The source explains: "Tests are the first layer of order we can leverage to define when and where we want a new lint to trigger or not." Writing tests first helps developers "find a balance for the first iteration of and further enhancements for a lint" and "not have to worry about over-engineering a lint on its first version nor missing out some obvious edge cases."

**Lintcheck** is a validation tool that builds and runs Clippy on a fixed set of real-world crates. After adding a new lint, running `cargo lintcheck` and auditing the results helps ensure there are no false positives and that suggestions are valid. The output can be `git diff`ed against the previous version to see the impact.

**Manual testing** is useful when `println!` debugging makes test output unreadable. Use `cargo dev lint input.rs` or `cargo dev lint /path/to/project` to run Clippy on arbitrary files or projects.

**MSRV testing**: When a lint has MSRV requirements, test cases should include both a version below the MSRV (lint should not trigger) and the MSRV version itself (lint should trigger):
```rust
#[clippy::msrv = "1.44"]
fn msrv_1_44() { /* should NOT trigger */ }

#[clippy::msrv = "1.45"]
fn msrv_1_45() { /* should trigger */ }
```

# Examples

**Example 1**: Test file for `foo_functions`:
```rust
#![warn(clippy::foo_functions)]

// Impl methods
struct A;
impl A {
    pub fn fo(&self) {}
    pub fn foo(&self) {}
    //~^ foo_functions
    pub fn food(&self) {}
}

// Plain functions
fn fo() {}
fn foo() {}
//~^ foo_functions
fn food() {}

fn main() {
    foo();
    let a = A;
    a.foo();
}
```

**Example 2**: Running tests during development:
```bash
# Run specific lint test
TESTNAME=foo_functions cargo uitest

# Generate .stderr reference files
TESTNAME=foo_functions cargo uibless

# Run full test suite
cargo test

# Run multiple specific tests
TESTNAME=foo_functions,bar_methods cargo uitest
```

**Example 3**: Expected `.stderr` output format:
```text
error: function called "foo"
  --> tests/ui/foo_functions.rs:6:12
   |
LL |     pub fn foo(&self) {}
   |            ^^^
   |
   = note: `-D clippy::foo-functions` implied by `-D warnings`
```

# Common Errors

- **Error**: Forgetting `#![warn(clippy::<lint_name>)]` at the top of the test file.
  **Correction**: Without this attribute, the lint may not be enabled in the test, leading to no output even after implementation.

- **Error**: Committing `.stderr` files changed by `cargo bless` for lints other than the one being developed.
  **Correction**: `cargo bless` may update more files than intended. Only commit files relevant to your specific lint.

- **Error**: Running `cargo uitest` instead of `cargo test` before submitting a PR.
  **Correction**: `cargo uitest` only runs UI tests. `cargo test` runs the full suite including dogfood tests that check Clippy does not lint itself.

# Common Confusions

- **Confusion**: Thinking you need to manually create `.stderr` files.
  **Clarification**: Never create `.stderr` files manually. Run `cargo bless` or `cargo uibless` to generate them automatically from Clippy's actual output.

- **Confusion**: Believing test failure during TDD means something is wrong.
  **Clarification**: Tests are written before implementation. Failure is expected until the lint logic emits diagnostics matching the `//~^` annotations. This is the normal TDD red/green cycle.

- **Confusion**: Wondering why `cargo lintcheck` is needed when UI tests already pass.
  **Clarification**: UI tests check expected behavior on contrived examples. Lintcheck validates against real-world crates to catch false positives and edge cases that contrived tests may miss.
