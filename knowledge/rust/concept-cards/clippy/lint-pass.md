# Quick Definition

A lint pass is an implementation of a trait (`EarlyLintPass` or `LateLintPass`) that uses the visitor pattern to traverse compiler data structures (AST or HIR), calling `check_*` methods on each node where the lint can inspect code and emit diagnostics.

# Core Definition

The source states: "We have to make this decision with every new Clippy lint. It boils down to using either `EarlyLintPass` or `LateLintPass`." This is the fundamental architectural choice in lint development.

Both pass types use the **visitor pattern**: the compiler walks the syntax tree and calls methods on the lint pass struct for each node. The lint author implements specific `check_*` methods (like `check_fn`, `check_expr`, `check_item`) to inspect nodes of interest. Each method receives a context object and the node being visited.

The two pass types differ in when they run during compilation:
- **EarlyLintPass**: Runs before type checking and HIR lowering. Operates on the AST. No access to type information.
- **LateLintPass**: Runs after type checking and HIR lowering. Operates on HIR. Has full access to type information via `LateContext`.

The `cargo dev new_lint` command defaults to `LateLintPass` because most lints need type information. You explicitly request `EarlyLintPass` with `--pass=early`.

A lint pass is registered with the compiler via `store.register_early_pass()` or `store.register_late_pass()`. Without registration, the pass will not run. Multiple lints can share a single pass (common in type-specific lint groups).

# Key Properties

1. `EarlyLintPass` runs before type checking; `LateLintPass` runs after type checking
2. `EarlyLintPass` operates on the AST (Abstract Syntax Tree); `LateLintPass` operates on HIR (High-Level Intermediate Representation)
3. The default for `cargo dev new_lint` is `LateLintPass` (the `--pass=late` default)
4. Both passes define `check_*` visitor methods: `check_fn`, `check_expr`, `check_item`, etc.
5. `check_expr` is by far the most common method because Rust is an expression language
6. Other common methods include `check_fn` and `check_item`
7. Registration order determines the order that lint output is emitted
8. Multiple lints can share a single lint pass (used by type-specific lint groups)
9. `EarlyLintPass` is faster since it runs before type checking, but this is not typically a concern
10. `EarlyLintPass` nodes are identified only by position in the AST; you cannot look up nodes by ID, making cross-node inspection harder

# Construction / Recognition

## To Choose a Lint Pass:
1. Does the lint need type information (e.g., checking if a type implements a trait, resolving method calls)? Use `LateLintPass`
2. Does the lint only inspect syntax (e.g., function names, code patterns, attributes)? Use `EarlyLintPass`
3. Does the lint need to inspect nodes other than the currently visited one by ID? Use `LateLintPass`
4. When in doubt, use `LateLintPass` -- it is the recommended default
5. If performance matters and no type info is needed, prefer `EarlyLintPass`

## To Implement a Lint Pass:
1. Define a struct for the lint pass (usually generated by `cargo dev new_lint`)
2. Implement the chosen trait (`EarlyLintPass` or `LateLintPass`) on the struct
3. Override specific `check_*` methods for the nodes you want to inspect
4. In each `check_*` method, apply your lint logic and call diagnostic functions
5. Register the pass with `store.register_early_pass()` or `store.register_late_pass()`

## To Recognize a Lint Pass in Code:
1. Look for `impl EarlyLintPass for <Struct>` or `impl LateLintPass for <Struct>`
2. Look for `declare_lint_pass!(<Struct> => [LINT_NAMES])` which generates the struct
3. Look for `check_*` methods inside the impl block

# Context & Application

Almost every Clippy lint is implemented as a `LateLintPass` because most useful checks require type information. The `EarlyLintPass` is reserved for lints that genuinely only care about syntax -- like checking function names, detecting certain syntactic patterns, or enforcing style rules that are visible in the AST alone.

The visitor pattern means lint authors do not control traversal. The compiler walks the entire tree and calls your `check_*` methods at each matching node. Your job is to filter within those methods (checking conditions, matching patterns) and emit a lint when conditions are met.

**Type-specific lint groups** (like `functions`, `methods`, `loops`) share a single lint pass. Individual lint logic lives in separate files but is called from the group's shared `check_*` methods in `mod.rs`. This is why `cargo dev new_lint --type=<type>` does not take a `--pass` flag -- the pass type is determined by the group.

# Examples

**Example 1**: EarlyLintPass implementation for `foo_functions`:
```rust
impl EarlyLintPass for FooFunctions {
    fn check_fn(&mut self, cx: &EarlyContext<'_>, fn_kind: FnKind<'_>, span: Span, _: NodeId) {
        if is_foo_fn(fn_kind) {
            span_lint_and_help(
                cx, FOO_FUNCTIONS, span,
                "function named `foo`",
                None,
                "consider using a more meaningful name"
            );
        }
    }
}
```

**Example 2**: LateLintPass implementation pattern:
```rust
impl<'tcx> LateLintPass<'tcx> for LintName {
    fn check_expr(&mut self, cx: &LateContext<'tcx>, expr: &'tcx Expr<'_>) {
        if some_lint_expr_logic(expr) {
            span_lint_and_help(
                cx, LINT_NAME, expr.span,
                "message on why the lint is emitted",
                None,
                "message that provides a helpful suggestion",
            );
        }
    }
}
```

**Example 3**: Registering a lint pass:
```rust
// In clippy_lints/src/lib.rs
store.register_early_pass(|| Box::new(foo_functions::FooFunctions));
// or
store.register_late_pass(|_| Box::new(foo_functions::FooFunctions));
```

# Common Errors

- **Error**: Implementing `EarlyLintPass` when the lint needs to check whether a type implements a trait.
  **Correction**: Use `LateLintPass` whenever type information is required. EarlyLintPass has no access to the type system.

- **Error**: Forgetting to register the lint pass after declaring it.
  **Correction**: Call `store.register_early_pass()` or `store.register_late_pass()` in `clippy_lints/src/lib.rs`. Without registration, the lint will silently not run.

- **Error**: Implementing the wrong `check_*` method for the nodes you want to inspect.
  **Correction**: Consult the `EarlyLintPass` or `LateLintPass` trait documentation to find the correct visitor method. For expressions use `check_expr`, for functions use `check_fn`, for items use `check_item`.

# Common Confusions

- **Confusion**: Thinking `EarlyLintPass` and `LateLintPass` are sequential stages of the same pipeline.
  **Clarification**: They are independent traits. A lint implements exactly one of them. Early passes run on the AST before HIR lowering; late passes run on HIR after type checking. They are different views of the code at different compilation stages.

- **Confusion**: Believing `check_expr` is only available on `LateLintPass`.
  **Clarification**: Both `EarlyLintPass` and `LateLintPass` have `check_expr` (and other `check_*` methods), but they receive different node types (AST nodes vs. HIR nodes) and different context types (EarlyContext vs. LateContext).

- **Confusion**: Assuming you need to implement tree traversal yourself.
  **Clarification**: The compiler handles traversal via the visitor pattern. You only implement the `check_*` methods for node types you care about. The compiler calls your methods as it walks the tree.
