# Quick Definition

Godoc uses specific formatting syntax: blank comment lines separate paragraphs, two-space indentation creates verbatim/code blocks, a standalone capitalized line followed by a paragraph creates a heading, and `Example` test functions produce runnable examples in documentation.

# Core Definition

Godoc provides specific syntax for formatting documentation. A blank comment line (`//`) separates paragraphs. Indenting lines by two additional spaces formats them verbatim (as code blocks), which can also be used for lists and tables. Test files can contain runnable examples (functions named `ExampleType_Method`) that appear attached to the corresponding documentation in godoc. A single line beginning with a capital letter, containing no punctuation except parentheses and commas, followed by another paragraph, is formatted as a heading with an autogenerated anchor tag. Preview documentation using the Go documentation server (`pkgsite`) both before and during code review.

# Construction / Recognition

## To Apply:
1. Use blank `//` lines between paragraphs
2. Indent code examples by two extra spaces in comments
3. Write `Example` test functions for interactive documentation
4. Format headings as a standalone capitalized line before a paragraph
5. Preview with `pkgsite` before submitting

## To Recognize:
1. Well-separated paragraphs in godoc output
2. Properly rendered code blocks from indented comment lines
3. Clickable headings with anchor links

# Context & Application

Godoc formatting is simpler than Markdown but has specific conventions that must be followed for proper rendering. Verbatim blocks (via indentation) are the primary way to include code in comments, but runnable examples are often more appropriate because they are compiled and tested. The heading syntax is strict -- any punctuation besides parentheses and commas will prevent heading rendering.

# Examples

**Example 1 -- Paragraphs**:

```go
// Good:
// LoadConfig reads a configuration out of the named file.
//
// See some/shortlink for config file format details.
```

**Example 2 -- Runnable example**:

```go
// Good:
func ExampleConfig_WriteTo() {
    cfg := &Config{
        Name: "example",
    }
    if err := cfg.WriteTo(os.Stdout); err != nil {
        log.Exitf("Failed to write config: %s", err)
    }
    // Output:
    // {
    //   "name": "example"
    // }
}
```

**Example 3 -- Verbatim code block**:

```go
// Good:
// Update runs the function in an atomic transaction.
//
// This is typically used with an anonymous TransactionFunc:
//
//   if err := db.Update(func(state *State) { state.Foo = bar }); err != nil {
//     //...
//   }
```

**Example 4 -- Heading**:

```go
// Good:
// The following line is formatted as a heading.
//
// Using headings
//
// Headings come with autogenerated anchor tags for easy linking.
```

# Common Errors

- **Error**: Forgetting the blank line between paragraphs
  **Correction**: Always use a blank `//` line to separate paragraphs in godoc

- **Error**: Including punctuation (periods, colons) in heading lines
  **Correction**: Headings must only contain capital letters, spaces, parentheses, and commas

# Common Confusions

- **Confusion**: Trying to use Markdown syntax in godoc comments
  **Clarification**: Godoc has its own formatting; Markdown does not render

- **Confusion**: Thinking code must be inline in comments
  **Clarification**: Runnable Example functions are often more appropriate and are compiled/tested
