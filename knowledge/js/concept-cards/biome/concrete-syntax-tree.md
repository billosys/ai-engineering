# Quick Definition
A Concrete Syntax Tree (CST) is a data structure similar to an Abstract Syntax Tree (AST) that preserves all information from the source code, including whitespace, tabs, and comments (collectively called trivia).

# Core Definition
The CST (Concrete Syntax Tree) is a data structure very similar to an AST (Abstract Syntax Tree) that keeps track of all the information of a program, trivia included. Biome's CST is implemented using an internal fork of the rowan library, which implements the Green and Red tree pattern. The CST is never directly accessible by design; developers read its information using the Red tree through autogenerated APIs derived from the language grammar.

# Prerequisites
- biome: Must understand what Biome is and its toolchain purpose

# Key Properties
1. **Complete information preservation** -- Retains all source code information including trivia (spaces, tabs, comments)
2. **Rowan-based implementation** -- Built on an internal fork of the rowan library
3. **Green and Red tree pattern** -- Uses this specific library pattern for tree representation
4. **Indirect access** -- Never directly accessible; accessed through the Red tree via autogenerated APIs
5. **Error-resilient construction** -- Requires a resilient and recoverable parser to construct
6. **Autogenerated API** -- Developer-facing APIs are autogenerated from the language grammar

# Construction / Recognition
A CST is constructed by Biome's resilient parser. The parser must be:
- **Resilient**: Able to resume parsing after encountering syntax errors
- **Recoverable**: Able to understand where an error occurred and resume creating correct information

The CST can represent both valid and invalid code, using bogus nodes to mark erroneous syntax.

# Context & Application
The CST is the central data structure underlying all of Biome's tools. Because it preserves all information including whitespace and comments, the formatter can make decisions about formatting while preserving comments. The linter can analyze the full syntactic structure. The CST's error resilience means Biome can process and provide feedback on code that contains syntax errors, which is essential for editor integration where code is frequently in an incomplete state.

# Examples
From internals/architecture.mdx, "Parser and CST" section:

Given this JavaScript snippet:
```js
const a = "foo"; // comment 1
// comment 2
const b = "bar";
```

The CST represents it as:
```
0: JS_MODULE@0..55
    ...
      1: SEMICOLON@15..27 ";" [] [Whitespace(" "), Comments("// comment 1")]
    1: JS_VARIABLE_STATEMENT@27..55
        ...
        1: CONST_KW@27..45 "const" [Newline("\n"), Comments("// comment 2"), Newline("\n")] [Whitespace(" ")]
  3: EOF@55..55 "" [] []
```

Note how `// comment 1` is trailing trivia on the semicolon, and `// comment 2` is leading trivia on the `const` keyword.

# Common Errors
1. Trying to access the CST directly -- it is accessed through the Red tree API, not directly
2. Assuming the CST cannot represent invalid code -- it can, using bogus nodes

# Common Confusions
1. **CST vs. AST** -- An AST abstracts away formatting details like whitespace and comments; a CST retains all of this information. Biome uses a CST specifically because tools like formatters need access to trivia.
2. **CST as a lossless representation** -- Unlike an AST, no information is lost when source code is parsed into a CST
